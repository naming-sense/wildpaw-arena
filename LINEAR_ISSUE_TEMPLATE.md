# LINEAR_ISSUE_TEMPLATE.md

Wildpaw Arena에서 Symphony가 잘 처리할 수 있게 Linear 이슈를 쓰는 **한국어 운영 기준 템플릿**입니다.

## 핵심 원칙

- **한 이슈 = 한 가지 목표**
- 되도록 **작은 diff**로 끝날 수 있게 쪼갤 것
- **수정 범위**와 **완료 조건**을 명확히 쓸 것
- 검증 명령을 꼭 적을 것
- 사람 판단이 필요한 일은 미리 명시하고 **`검토 필요`에서 멈추게** 할 것

## 상태 추천

- 아직 준비 안 됨 → `백로그`
- 바로 맡겨도 됨 → `할 일`
- 이미 누가 작업 중 / Symphony가 잡아감 → `진행 중`
- 사람 확인 필요 → `검토 필요`
- 끝남 → `완료`
- 취소 → `취소`
- 중복 → `중복`

> 현재 Symphony는 Wildpaw 프로젝트에서 `할 일`, `진행 중` 상태 이슈를 대상으로 동작합니다.

## 제목 작성 규칙

권장 형식:
- **`[영역] 무엇을 바꾸는지`**
- 제목은 되도록 **한국어로**, 짧고 구체적으로 쓴다
- 영역(prefix)도 한국어를 기본값으로 쓴다

권장 영역 예시:
- `[하네스]`
- `[게이트웨이]`
- `[클라이언트]`
- `[프로토콜]`
- `[레벨]`
- `[문서]`
- `[서버]`

좋은 제목 예시:
- `[게이트웨이] smoke 재시도 시 포트 충돌 로그 개선`
- `[클라이언트] level validate 경고 메시지 포맷 정리`
- `[프로토콜] hello ack 필드 문서화 및 generated sync`
- `[하네스] check.sh에서 protocol drift 출력 개선`
- `[문서] Symphony 운영 런북 추가`

피해야 할 제목:
- `버그 수정`
- `이거 좀 개선`
- `전체적으로 정리`
- `알아서 좋은 방향으로 정리`

# Symphony가 좋아하는 이슈 / 싫어하는 이슈

## 좋아하는 이슈
- 범위가 작다
- 수정 경로가 보인다
- 완료 조건이 명확하다
- 검증 명령이 있다
- 사람 취향 판단이 적다

## 싫어하는 이슈
- “전체적으로 개선”
- “알아서 제일 좋은 방향으로”
- 기획/밸런스/감성 판단이 핵심
- 완료 조건이 없다
- 수정 금지 경계가 없다

# Wildpaw용 추천 규칙

- 작은 작업은 **Tier A**로 쓰기
- 멀티 서피스 변경은 **Tier B**로 표시하기
- 큰 방향성/기획 변경은 **Tier C**로 분리하기
- 사람 확인이 필요하면 완료 상태를 곧바로 `완료`로 하지 말고 **`검토 필요`에서 멈추게 쓰기**
- 설명 마지막에 검증 명령을 빼먹지 말기
- 상태는 가능하면 `백로그 → 할 일 → 진행 중 → 검토 필요 → 완료` 흐름으로 쓰기
