# 24. 매칭 실패 대체봇 RL 설계서 (MDP · 리워드 · 학습 · 시뮬레이션)

## 0) 문서 목적
PvP 큐 대기 중 매칭이 지연/실패될 때, 유저가 이탈하지 않도록 **강화학습(RL) 기반 대체봇**을 투입하는 전체 설계를 정의한다.

본 문서는 아래를 고정한다.
1. 서비스 플로우(큐 타임아웃 → 봇 매치 전환)
2. RL 문제정의(MDP/POMDP)와 액션/관측 설계
3. 리워드 함수 및 리워드 해킹 방지 규칙
4. 학습 파이프라인(모사학습 + self-play RL)
5. 시뮬레이터 구조(권위형 C++ 룸서버 재사용)
6. 배포/운영/평가/KPI 및 롤아웃 계획

## 1) 문제 정의 및 제품 목표

## 1-1. 현재 문제
- 특정 시간대/리전에서 PvP 큐 대기시간이 길어져 이탈률 상승
- 초반 체류(첫 3~10판)에서 "매칭이 안 잡힘"이 가장 큰 포기 요인

## 1-2. 목표
1. **큐 이탈률 감소**: 장기 대기 유저를 즉시 플레이로 전환
2. **경기 품질 유지**: 봇 매치라도 손맛/전술/난이도 체감 유지
3. **공정성 보존**: 랭크 신뢰도 손상 방지
4. **운영 가능성**: 모델 업데이트/롤백/모니터링이 쉬운 파이프라인

## 1-3. 적용 범위(권장)
- **Phase 1 (MVP): 일반 큐(3v3/5v5)만 적용**
- **Phase 2:** 신규 보호 매치 + 이벤트 큐 확장
- **Phase 3:** 랭크는 제한적 적용(오프피크 + 명시 동의 + RP 보정)

> 랭크는 경쟁 신뢰가 핵심이므로, 초기에는 일반 큐 중심이 안전하다.

## 2) 매칭 플로우 설계 (봇 대체 진입)

## 2-1. 큐 타임아웃 정책
모드별 1차 권장값:
- 3v3 일반: `T_bot_offer = 45s`
- 5v5 일반: `T_bot_offer = 60s`
- 신규 보호 풀: `T_bot_offer = 25s`
- 랭크: MVP 비활성(또는 120s 이후 동의형)

## 2-2. 유저 선택 UX
타임아웃 시 팝업:
- 옵션 A: `지금 시작(봇 포함)`
- 옵션 B: `사람 매칭 계속 대기`

표시 문구:
- "현재 매칭 인원이 부족해 AI 플레이어가 포함될 수 있습니다"
- 예상 대기시간 비교(즉시 시작 vs 계속 대기)

## 2-3. 파티 정책
- 파티 리더가 선택, 멤버 전원에게 상태 동기화
- 파티 MMR 평균 기준으로 봇 난이도 산정

## 2-4. 매치 생성 규칙
- 부족 슬롯 수(`n_missing`)만큼 봇 채움
- 봇 팀 구성은 역할군 결손 우선 보정(탱/딜/컨트롤)
- 1인만 인간인 완전 PvE 느낌 매치는 회피(최소 인간 팀원 비율 유지)

## 3) RL 문제정의: MDP/POMDP 모델

탑뷰 팀 슈터는 부분관측 다중에이전트 환경이므로 실제론 **POMDP 기반 Markov Game**으로 본다.
실구현은 중앙집중 critic이 있는 MAPPO 구조를 기준으로 한다.

## 3-1. 시간 단위
- 서버 틱: 30Hz (authoritative)
- 정책 결정 주기: 5Hz (`200ms`마다 의사결정)
- 5Hz 사이 프레임은 로코모션/에임 보간 + 룰기반 보조

## 3-2. 상태(관측) 벡터 `o_t`

### A) 자기 상태
- 위치/속도/방향
- HP/실드/탄약/재장전 상태
- 스킬 쿨다운(Q/E/R), 궁극기 충전
- 상태이상(슬로우/스턴/은신 등)

### B) 아군 정보(상대좌표 기준)
- 각 아군의 상대 위치, HP%, 궁극기 준비 여부
- 최근 2초 전투참여 플래그(교전중/이탈)

### C) 적 정보 (가시 + 기억)
- 시야 내 적: 위치/HP 추정/위협도
- 비가시 적: 마지막 목격 시점/위치(메모리 채널)

### D) 맵/오브젝트 상태
- 점령 게이지, 오브젝트 리스폰 타이머
- 엄폐물 밀집도/위험 구역 히트맵

### E) 매치 컨텍스트
- 남은 시간, 점수차, 팀 조합 타입
- 현재 하이레벨 택틱 상태(푸시/리그룹/후퇴)

## 3-3. 액션 공간 `a_t` (하이브리드)

### 하이레벨(1Hz)
- `ENGAGE_TARGET(k)`
- `HOLD_OBJECTIVE(j)`
- `RETREAT(anchor)`
- `REGROUP(ally_id)`
- `FLANK(route_id)`

### 로우레벨(5Hz)
- 이동 벡터 `(dx, dy)`
- 조준 각도 `theta`
- 발사 `fire ∈ {0,1}`
- 스킬 사용 `useQ/useE/useR`
- 재장전/상호작용

> 구현 단순화를 위해 초기엔 액션을 이산화(이동 8방향+정지, 조준 16분할) 후 점진적으로 연속화 가능.

## 3-4. 전이/종료
- 전이 `P(s_{t+1}|s_t,a_t)`는 룸 서버 물리/전투 규칙으로 결정
- 종료 조건:
  - 경기 시간 만료
  - 목표 점수 달성
  - 조기 종료(연결 실패 등 운영 규칙)

## 4) 리워드 설계

## 4-1. 기본 형태
팀게임 특성상 팀 리워드를 중심으로 설계:

`r_i = α * r_team + (1-α) * r_individual`
- 권장: `α = 0.75`

총 리워드:

`r_total = w_obj*r_obj + w_fight*r_fight + w_surv*r_surv + w_macro*r_macro - w_penalty*r_penalty`

## 4-2. 이벤트별 리워드 예시(초기값)
| 항목 | 값 | 의도 |
|---|---:|---|
| 오브젝트 점령/기여 | +1.2 | 승리 목적 행동 강화 |
| 킬 관여(킬/어시) | +0.6 | 팀 교전 기여 강화 |
| 사망 | -0.7 | 무리한 진입 억제 |
| 아군 보호/커버(피해 분산) | +0.3 | 팀플 강화 |
| 무의미한 단독 돌진 | -0.25 | 트롤성 행동 억제 |
| 궁극기 낭비(가치 없는 사용) | -0.2 | 스킬 효율 학습 |
| 전투 없는 장기 배회(10초+) | -0.15 | 목적 없는 이동 억제 |
| 승리 보너스(종료시) | +2.0 | 최종 목표 정렬 |
| 패배 페널티(종료시) | -2.0 | 결과 반영 |

## 4-3. 잠재함수 기반 shaping (권장)
리워드 치팅을 줄이기 위해 Potential-based shaping 사용:

`F(s,s') = γΦ(s') - Φ(s)`

`Φ(s)` 예시:
- 승리확률 추정치(점수차/오브젝트 상태/생존자 수 기반)

이 방식은 최적정책 보존 성질을 가져 메인 목표 왜곡을 줄인다.

## 4-4. 리워드 해킹 방지
- 단일 지표 과최적화 방지(킬만 추구 금지)
- 벽 사이 사격/비정상 시야 활용 패널티
- 행동 entropy 하한 유지(단조로운 루프 방지)
- 수치 변경은 `bot_reward_weights_vX.json`으로 버전관리

## 5) 학습 전략

## 5-1. 알고리즘 권장
- 기본: **MAPPO (Centralized Critic + Decentralized Actor)**
- 부분관측 대응: GRU/LSTM policy
- 안정화: reward normalization, value clipping, GAE

## 5-2. 3단계 학습 파이프라인

### Stage A. 부트스트랩(규칙 + 모사학습)
1. 룰기반 봇으로 초기 데이터 생성
2. 인간 리플레이/내부 플레이 로그로 행동클로닝(BC)
3. 초기 정책 품질 확보(무작위 행동 구간 최소화)

### Stage B. Self-Play RL
1. 현재 정책 vs 과거 정책(snapshot pool) 대전
2. 맵/조합 랜덤화로 일반화
3. 난이도별 정책 분화(초급/중급/상급)

### Stage C. 리그 학습(League Training)
- 메인 정책, 익스플로잇 정책, 카운터 정책 동시 운영
- 특정 메타/영웅 조합 과적응 방지

## 5-3. 커리큘럼
1. 1v1 교전 미니게임(에임/거리조절)
2. 3v3 소맵 오브젝트 모드
3. 5v5 풀룰 + 드래프트 변형
4. 실제 라이브 로그 기반 시나리오 리플레이

## 6) 시뮬레이션/학습 인프라 설계

## 6-1. 핵심 원칙
- **렌더링 없는 Headless authoritative 시뮬레이터** 사용
- 클라이언트가 아닌 서버 규칙으로 학습(실전 일치)

## 6-2. 구성
1. **Env Worker (C++)**
   - room 서버 로직 재사용
   - step/reset 인터페이스 제공
2. **Trajectory Collector**
   - 관측/행동/보상/종료 수집
3. **Learner (Python/Torch)**
   - MAPPO 업데이트 수행
4. **Model Registry**
   - `bot_policy_{mode}_{version}.onnx` 저장/배포
5. **Evaluator**
   - 기준 봇/인간 리플레이 상대 성능 측정

## 6-3. 처리량 목표(초기)
- 동시 env: 128~256
- 샘플 수집: 50k~150k step/min (하드웨어 의존)
- 학습 주기: 6~12시간 단위 체크포인트

## 6-4. 도메인 랜덤화
- 반응속도 노이즈(120~260ms)
- 에임 미세 오차
- 맵 스폰/오브젝트 타이밍 편차
- 조합 랜덤(영웅/로드아웃)

> 완벽 조준/완벽 반응 금지. “사람처럼 강한” 목표를 유지.

## 7) 런타임 배포 설계 (실서버)

## 7-1. 추론 위치
- **서버 추론 고정** (권장)
  - 장점: 치트/모델 유출 위험 감소, 동기화 일관성

## 7-2. 의사결정 예산
- 1봇당 추론 예산: `<= 1.5ms` (CPU 기준 목표)
- 예산 초과 시 폴백:
  1. 이전 액션 유지
  2. 규칙기반 안전행동(엄폐/리그룹)

## 7-3. 난이도 매핑
- 입력: 큐 MMR, 파티 크기, 신규/복귀 여부
- 출력: BotTier(S1/S2/S3), 반응속도, 에임분산

예시:
- 저 MMR: 반응 느림 + 궁극기 사용 빈도 감소
- 고 MMR: 팀 합류 속도/오브젝트 판단 강화

## 10) 평가 지표(KPI) 및 승격 게이트

## 10-1. 서비스 KPI
- 큐 이탈률 감소폭
- 평균 큐 대기시간 감소
- 봇 매치 수락률
- 봇 매치 이후 24h 재방문율

## 10-2. 경기 품질 KPI
- 인간팀 승률 편차(목표: 45~55%)
- 1킬 시간/평균 교전 길이/오브젝트 기여도
- 봇 비정상 행동률(벽끼임/무한 배회/AFK 유사)

## 10-3. ML 운영 KPI
- 추론 p95 latency
- 버전별 승률 드리프트
- 패치 후 성능 회귀율

## 10-4. 모델 승격 조건(예시)
1. 기준봇 대비 승률 +3% 이상
2. 인간 상대 체감평가(내부 QA) 4.0/5 이상
3. 비정상 행동률 0.5% 미만
4. 추론 p95 목표치 충족

## 11) 롤아웃 계획 (8주 예시)

### W1~W2
- 환경 계측, 학습 인터페이스(step/reset) 구축
- 룰기반 baseline + 데이터 파이프라인 정리

### W3~W4
- BC 초기모델 + MAPPO 1차 학습
- 3v3 일반 내부 테스트

### W5
- A/B(5%) 시작: 봇옵션 노출, KPI 모니터링

### W6
- 리워드/난이도 튜닝, 비정상 행동 제거

### W7
- A/B(20~30%) 확대, 신규 유저 풀 적용

### W8
- Phase 1 정식 오픈 판단(Go/No-Go)

## 12) 리스크 및 대응
| 리스크 | 설명 | 대응 |
|---|---|---|
| 봇이 너무 강함 | 초보 유저 좌절 | 난이도 캡, 반응속도 상한, 신규 보호 프로필 |
| 봇이 너무 약함 | 재미 저하, 학습효과 부족 | MMR 기반 강도 매핑, 팀 보상 강화 |
| 반복 패턴 노출 | 인간이 쉽게 악용 | 정책 풀 로테이션, 탐색 노이즈 |
| 리워드 해킹 | 목표와 다른 행동 최적화 | shaping 검증, 오프라인 평가군 다변화 |
| 서버 비용 상승 | 추론/시뮬레이션 부하 | 배치 추론, 결정주기 최적화, 폴백 규칙 |

## 14) 최종 의사결정 제안
1. **지금 바로 착수할 범위:** 3v3 일반 + 동의형 봇 대체
