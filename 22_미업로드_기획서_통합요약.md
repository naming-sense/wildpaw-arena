# 22. 미업로드 기획서 통합 요약 (18~21)

## 0) 문서 목적
최근 작성했지만 아직 리포지토리에 반영되지 않았던 기획서들을
한 번에 이해할 수 있도록 핵심만 요약한다.

대상 문서:
- `18_캐릭터_무기_전투_클라이언트_개발계획서.md`
- `19_게임플로우_온보딩_로비_매칭_드래프트_UXUI_상세.md`
- `20_게임플로우_API_이벤트_서버계약서.md`
- `21_UXUI_화면별_컴포넌트_상세명세서.md`

---

## 1) 핵심 배경
문제: 현재 프로토타입은 실행 즉시 전투 진입이라,
상용 서비스에 필요한 "사용자 흐름"과 "제어 이벤트 계약", "UI 컴포넌트 구현 기준"이 부족했다.

해결: 18~21 문서로
1) 전투 구현 계획,
2) 전체 게임플로우,
3) 서버/클라 이벤트 계약,
4) 화면 단위 UI 명세
를 연결해 개발 공백을 메웠다.

---

## 2) 문서별 요약

## 2-1. 18번 요약 — 전투 클라이언트 개발계획
목표:
- 02 전투디자인을 **클라 개발 가능한 스펙**으로 변환

핵심 내용:
- 데이터 SSoT(`hero_balance_mvp_v0.2.json`) 기준 설계
- Hero/Weapon/Skill 타입 및 validation 규칙
- 무기/스킬/상태이상 로직 구조
- 예측/리컨실리에이션 연동
- 4주 스프린트 + QA 체크리스트

성과:
- "컨셉"을 "구현 계획"으로 전환

## 2-2. 19번 요약 — 게임플로우 + UX/UI 상세
목표:
- 처음 실행부터 결과 화면까지 상태 전이 구멍 제거

핵심 내용:
- 상태 머신 정의(BOOT→AUTH→ONBOARDING→LOBBY→QUEUEING→DRAFT→IN_MATCH→RESULT)
- 방 생성(자동/커스텀), 큐/레디체크/드래프트 흐름
- 실패/예외 처리(재접속, 할당 실패, 타임아웃)
- 화면별 UX 와이어프레임 수준 정의

성과:
- "지금 왜 이 화면인지"가 항상 설명되는 서비스 플로우 확보

## 2-3. 20번 요약 — API 이벤트 서버 계약
목표:
- 19번 플로우를 서버/클라 공통 프로토콜로 고정

핵심 내용:
- Control channel 이벤트(C2S/S2C) 스키마
- payload 구조, timeout/retry, 에러 코드
- 멱등성(eventId/requestId) 규약
- Match assign 이후 room realtime으로 경계 전환

성과:
- 구현/QA/로그 분석이 가능한 이벤트 계약 확정

## 2-4. 21번 요약 — 화면별 컴포넌트 명세
목표:
- UI를 화면 단위에서 바로 개발할 수 있는 수준으로 명세

핵심 내용:
- 공통 컴포넌트(Button/Modal/Toast/Badge)
- 상태별 컴포넌트 가시성 규칙
- 화면별 바인딩 데이터/이벤트
- 접근성, 반응형, 로컬라이징, QA 기준

성과:
- 디자이너-클라개발-QA가 같은 기준으로 작업 가능

---

## 3) 이번 업로드의 실제 가치

1. **전투 구현**과 **게임플로우**가 분리되지 않고 연결됨
2. UX 문서가 추상 가이드가 아니라 이벤트 계약까지 이어짐
3. 기획 변경 시 수정 우선순위가 명확해짐
   - 플로우 변경: 19
   - API 계약 변경: 20
   - UI 구현 변경: 21
   - 전투 수치/로직 변경: 18

---

## 4) 다음 액션 권장

1. `19` 상태 머신을 클라이언트 전역 store에 반영
2. `20` 이벤트를 Gateway 스펙으로 1:1 매핑
3. `21` 컴포넌트를 스토리북으로 상태별 시각 검증
4. `18` 기준으로 전투 QA 자동 시나리오(회귀) 구축

---

## 5) 결론
18~21은 각각 따로 있는 문서가 아니라,
**전투 구현(18) → 서비스 흐름(19) → API 계약(20) → UI 구현(21)**로 이어지는 하나의 실행 패키지다.

이 업로드로 "기획은 있는데 구현 기준이 없음" 상태를 해소하고,
실제 개발 착수 가능한 기준선을 확보했다.
