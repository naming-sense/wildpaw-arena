# AI를 활용한 서버/클라이언트 개발 가이드라인 - 사용 케이스 / 적용 예시

이 문서는 `ai-game-dev-guidelines.md`의 보조 문서다.  
메인 문서가 핵심 원칙 요약본이라면, 이 문서는 **실제 업무에서 어떻게 적용하는지**를 예시 중심으로 보여준다.

---

## 1. Case A 사용 케이스

## A-1. 클라이언트 전투 감각 검증

### 상황
새로운 이동/대시/피격 감각을 빠르게 확인하고 싶다.  
아직 메인 프로젝트 구조에 넣을 단계는 아니다.

### 목표
- 실제 조작이 가능한 최소 전투 루프를 만든다.
- 수치를 빠르게 바꾸며 감각을 비교한다.
- 구현보다 검증 속도를 우선한다.

### AI에 맡길 것
- 전투 프로토타입 모듈
- 입력 시퀀스를 반복할 수 있는 하네스
- 수치 조정 포인트 정리
- 로그/화면 관찰 포인트 추가

### 결과물
- 플레이 가능한 테스트 장면
- 대시/피격/쿨다운 검증용 하네스
- 어떤 감각이 좋았는지에 대한 기록
- B 단계로 옮길 때 필요한 Migration Notes

---

## A-2. 서버 룸 상태 전이 검증

### 상황
룸 생성 → 입장 → 시작 → 종료 흐름이 성립하는지 빨리 보고 싶다.

### 목표
- 네트워크 전체를 다 붙이지 않고 상태 전이만 먼저 검증한다.
- 이벤트 순서와 실패 케이스를 확인한다.

### AI에 맡길 것
- 고정 시나리오 runner
- mock 세션 / mock 유저
- 상태 전이 로그 출력
- 성공/실패 케이스 시뮬레이터

### 결과물
- 세션 상태 전이 하네스
- 정상/예외 시나리오 로그
- 실제 메인 프로젝트에 넣기 전 확인된 위험 포인트

---

## A-3. 패킷/이벤트 순서 검증

### 상황
클라이언트 UI가 특정 이벤트 순서에 의존한다.  
서버 응답 순서가 바뀌면 UI가 꼬일 수 있다.

### 목표
- 실제 멀티플레이 전체를 띄우지 않고 이벤트 순서를 검증한다.
- replay 가능한 시나리오를 만든다.

### AI에 맡길 것
- packet replay harness
- 이벤트 순서 기록/재생 도구
- UI 반응 검증용 mock 상태

### 결과물
- 재현 가능한 packet/event replay 세트
- 어떤 이벤트 순서에서 문제가 나는지에 대한 기록
- B에서 계약 문서화할 포인트

---

## 2. Case B 사용 케이스

## B-1. A에서 만든 모듈을 메인 프로젝트로 옮기기

### 상황
프로토타입에서 검증한 모듈이 괜찮아서, 이제 메인 프로젝트에 넣어야 한다.

### 권장 순서
1. 관련 기능의 기존 코드베이스를 분석한다.
2. 프로젝트의 코딩 컨벤션과 구조 규칙을 정리한다.
3. A 모듈과 현재 구조의 차이를 Gap 분석한다.
4. 마이그레이션 전략 / 요구사항 / 개발 계획 문서를 쓴다.
5. 작은 단위로 구현한다.
6. 코드 리뷰를 진행한다.
7. 테스트 프레임워크와 실제 테스트를 붙인다.
8. 실패 케이스를 다시 수정하고 재검증한다.

### AI에 맡길 것
- 코드베이스 분석 문서 초안
- 컨벤션 요약
- Gap 분석표
- Migration Plan 초안
- 단계별 구현 보조
- 리뷰 포인트 정리
- 테스트 포인트 제안

### 핵심 포인트
프로토타입 코드를 그대로 복붙하지 말고,
**무엇을 유지하고 무엇을 버릴지 먼저 문서화**해야 한다.

---

## B-2. 기존 프로젝트에 새 기능을 협업 방식으로 추가하기

### 상황
기존 프로젝트에 서버/클라이언트 기능을 추가해야 하고, 다른 개발자와 같이 작업한다.

### 목표
- 기존 구조를 깨지 않고 기능을 붙인다.
- 이후 다른 개발자가 이어서 작업할 수 있게 만든다.

### 권장 순서
1. 기능별 진입점과 영향 범위를 파악한다.
2. 기존 구조에서 어디에 붙어야 하는지 정한다.
3. 변경 계약과 상태 전이 규칙을 먼저 정리한다.
4. 구현 계획을 문서화한다.
5. 작은 PR 단위로 구현한다.
6. 리뷰 후 테스트를 붙인다.

### AI에 맡길 것
- 진입점 분석
- 영향 범위 요약
- 계약 초안 작성
- 작은 변경 단위 분해
- 리뷰 체크리스트 생성

---

## B-3. 테스트 프레임워크가 약한 프로젝트에서 AI 활용

### 상황
프로젝트에 테스트 체계가 약해서 구현 후 매번 수동 확인만 하고 있다.

### 목표
- 최소한의 자동 테스트 기반을 만든다.
- 이후 반복 작업 비용을 낮춘다.

### AI에 맡길 것
- 현재 구조에 맞는 테스트 진입점 제안
- fixture / mock / helper 생성
- smoke test 초안 작성
- contract test 초안 작성
- 테스트 실패 로그 요약 보조

### 핵심 포인트
B에서는 기능 구현만큼,
**반복 가능한 테스트 체계를 붙이는 작업 자체도 별도 업무**로 봐야 한다.

## 3. 실무에서 AI를 운영하는 추천 방식

실제 업무에서는 AI에게 한 번에 너무 많은 역할을 맡기면 품질이 흔들리기 쉽다.  
아래처럼 턴을 나눠서 쓰는 방식이 더 안정적이다.

### 1) 분석 턴
- 기능별 코드베이스 분석만 시킨다.

### 2) 규칙 추출 턴
- 코딩 컨벤션, 구조 규칙, 안티패턴만 정리시킨다.

### 3) 계획 턴
- 마이그레이션 전략 / 개발 계획 문서만 작성시킨다.

### 4) 구현 턴
- 문서 기준으로 작은 단위 구현만 시킨다.

### 5) 리뷰 턴
- 구조 위반, 예외 처리 누락, 테스트 누락 등을 점검시킨다.

### 6) 테스트 턴
- 테스트 케이스, fixture, smoke test, contract test를 설계시킨다.

### 7) 이터레이션 턴
- 실패 로그 요약, 원인 가설, 다음 수정 우선순위만 정리시킨다.

## 4. 바로 쓸 수 있는 요청 예시

## A용 요청 예시
```md
목표:
- 전투 대시 감각을 빠르게 검증한다.

요청:
- 플레이 가능한 최소 모듈을 만든다.
- 입력 시퀀스를 반복 실행할 수 있는 하네스를 함께 만든다.
- 주요 수치를 쉽게 바꿀 수 있게 한다.
- 결과를 로그와 화면에서 확인할 수 있게 한다.

제외:
- 메인 프로젝트 통합
- 최종 아키텍처 정리
- 실제 멀티플레이 전체 연결
```

## B용 요청 예시
```md
목표:
- 프로토타입에서 검증된 룸 상태 전이 모듈을 메인 프로젝트에 편입한다.

먼저 할 일:
1. 관련 기능 코드베이스를 분석한다.
2. 코딩 컨벤션과 구조 규칙을 정리한다.
3. A 모듈과 현재 구조의 Gap을 분석한다.
4. Migration Plan 문서를 작성한다.

구현 시 요구사항:
- 작은 단위로 구현할 것
- 변경 이유와 영향 범위를 설명할 것
- 테스트 포인트를 제안할 것
- 리뷰 가능한 형태로 정리할 것
```

## 5. 마지막 정리

이 문서의 실무적 메시지는 명확하다.

- A에서는 AI에게 **실험용 하네스와 프로토타입**을 빨리 만들게 한다.
- B에서는 AI를 **분석, 문서화, 구현, 리뷰, 테스트를 돕는 협업 도구**로 쓴다.

즉 AI를 잘 쓰는 핵심은,
한 번에 모든 걸 맡기는 것이 아니라 **단계별 역할을 나눠서 운영하는 것**이다.
