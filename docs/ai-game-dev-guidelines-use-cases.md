# AI를 활용한 서버/클라이언트 개발 가이드라인 - 사용 케이스 / 적용 예시

이 문서는 `ai-game-dev-guidelines.md`의 보조 문서다.  
메인 문서가 핵심 워크플로우 요약본이라면, 이 문서는 **실제 업무에서 A와 B가 어떻게 다르게 굴러가는지**를 예시 중심으로 설명한다.

---

## 1. 먼저 이해해야 할 포인트

A와 B는 공통 실행 루프를 공유한다.

1. 요구사항 / 개발 계획 문서화
2. 문서 기반의 작은 단위 구현
3. AI 리뷰 + 인간 리뷰
4. 테스트 준비
5. 실제 테스트 수행
6. 문제 발생 시 반복 이터레이션

차이는 마지막 반복 이터레이션에서 난다.

- **A는 문서/하네스/시나리오를 기준으로 수정**한다.
- **B는 코드/구조/테스트를 기준으로 수정**한다.

아래 예시들은 이 차이를 보여주기 위한 것이다.

---

## 2. 같은 문제를 A와 B에서 다르게 푸는 예시

## 예시 1. 결과 UI가 서버 확정 전에 먼저 뜨는 문제

### 상황
클라이언트에서 결과 UI가 서버 확정 이벤트보다 먼저 떠서 흐름이 어색하다.

### A에서 푸는 방식
A에서는 이 문제를 **행동 결과 기준 문제**로 본다.

#### 접근
- 요구사항 문서에 “결과 UI는 서버 확정 응답 이후에만 표시”를 명시한다.
- 하네스 시나리오에 이벤트 순서 검증 케이스를 추가한다.
- AI에게 수정된 요구사항 기준으로 모듈과 하네스를 다시 맞추게 한다.
- 다시 실행해서 결과 흐름을 본다.

#### 포인트
- 코드를 세밀하게 해부하는 것이 핵심이 아니다.
- **원하는 동작과 검증 시나리오를 더 정확하게 만드는 것**이 핵심이다.

### B에서 푸는 방식
B에서는 이 문제를 **상태 전이 / 이벤트 처리 코드 문제**로 본다.

#### 접근
- 결과 UI를 띄우는 코드 경로를 찾는다.
- 서버 확정 이벤트와 클라이언트 상태 갱신 순서를 확인한다.
- 필요하면 계약 문서와 상태 전이 문서를 수정한다.
- 실제 코드 패치를 하고 리뷰를 거친다.
- contract test 또는 UI smoke test를 추가한다.

#### 포인트
- B에서는 문제를 **실제 코드 경로와 계약 수준**에서 해결해야 한다.

## 예시 2. 룸 종료 후 로비 복귀가 가끔 실패하는 문제

### A에서 푸는 방식
- 종료 시나리오를 하네스에 추가한다.
- 어떤 순서에서 실패하는지 로그를 더 본다.
- 요구사항을 “종료 후 반드시 로비 복귀 상태가 된다”로 더 명확히 적는다.
- mock 세션/종료 이벤트를 조정해서 다시 생성하고 재검증한다.

### B에서 푸는 방식
- 세션 종료 처리 경로와 상태 전이 코드를 직접 본다.
- cleanup 누락, 이벤트 순서, 상태 초기화 타이밍을 점검한다.
- 필요한 코드 패치를 한 뒤 테스트를 추가한다.
- 반복 실패 시 관련 구조 자체를 재설계한다.

---

## 예시 3. 특정 패킷 순서에서 UI가 꼬이는 문제

### A에서 푸는 방식
- packet replay harness를 만든다.
- 문제 패턴을 재현 가능한 시나리오로 바꾼다.
- AI에게 replay 기준으로 결과를 맞추게 한다.
- 검증 후 “이 패킷 순서는 위험”이라는 메모를 남긴다.

### B에서 푸는 방식
- 패킷 처리 핸들러, 큐, 상태 갱신 순서를 본다.
- 클라이언트/서버 계약을 수정하거나 정교화한다.
- 테스트 프레임워크 안에 replay/contract test를 편입한다.
- 이후 회귀 테스트로 관리한다.

---

## 3. A에서 AI를 쓰는 실제 감각

A에서는 AI에게 최종 코드 자산을 요구하지 않는다.  
대신 다음을 잘 시키는 것이 중요하다.

- 빠르게 돌릴 수 있는 하네스 만들기
- mock/stub 데이터 만들기
- 입력 시나리오 정리하기
- 결과를 눈에 보이게 로그/화면에 드러내기
- 요구사항 기준으로 다시 생성하기

즉 A에서 AI는 **검증 속도를 높이는 도구**다.

### A에서 좋은 요청 방식
- “이 시나리오에서 원하는 결과가 나오도록 다시 맞춰줘”
- “이 요구사항을 반영해서 하네스까지 같이 수정해줘”
- “이 입력 시퀀스로 반복 검증 가능하게 만들어줘”

### A에서 피해야 할 방식
- “대충 더 좋게 바꿔줘”
- “이 코드 왜 이래?”
- “메인 프로젝트 넣을 수준으로 다 정리해줘”

---

## 4. B에서 AI를 쓰는 실제 감각

B에서는 AI를 단순 생성기가 아니라,
**분석 + 계획 + 구현 + 리뷰 + 테스트 보조 도구**로 써야 한다.

### B에서 먼저 시킬 것
1. 기능별 코드베이스 분석
2. 코딩 컨벤션 / 구조 규칙 정리
3. Gap 분석
4. 마이그레이션 전략 문서화

### B에서 구현 단계에서 시킬 것
- 작은 단위 구현
- 변경 이유 설명
- 영향 범위 설명
- 테스트 포인트 제안
- 리뷰 포인트 정리

### B에서 문제 발생 시 시킬 것
- 실패 로그 요약
- 원인 후보 정리
- 수정 우선순위 정리
- 회귀 가능 구간 표시
- 테스트 보강 포인트 정리

즉 B에서 AI는 **코드 생성기**보다는 **개발 파이프라인 가속기**에 더 가깝다.

---

## 5. 실무에서 가장 유용한 운영 방식

실제로는 AI에게 모든 걸 한 번에 맡기기보다,
아래처럼 턴을 나눠 쓰는 방식이 더 잘 맞는다.

### 1) 분석 턴
- 관련 코드와 구조만 분석시킨다.

### 2) 계획 턴
- 요구사항 / 개발 계획 / 마이그레이션 문서만 쓰게 한다.

### 3) 구현 턴
- 문서를 기준으로 작은 단위만 구현하게 한다.

### 4) 리뷰 턴
- 구조 위반, 컨벤션 위반, 테스트 누락을 점검하게 한다.

### 5) 테스트 턴
- 하네스, smoke test, contract test, fixture를 정리하게 한다.

### 6) 이터레이션 턴
- 실패 원인과 다음 수정 우선순위만 정리하게 한다.

이렇게 나누면 A든 B든 품질이 훨씬 안정적이다.

---

## 6. 마지막 정리

이 문서의 핵심은 아주 단순하다.

- A와 B는 공통 실행 루프를 공유한다.
- 차이는 **문제가 생겼을 때 무엇을 기준으로 고치느냐**다.

즉,

- **A는 문서/하네스 중심 반복**
- **B는 코드/구조/테스트 중심 반복**

이 관점으로 보면 AI는 단순 코더가 아니라,
단계별로 다르게 써야 하는 **업무 운영 도구**가 된다.
