# AI를 활용한 서버/클라이언트 개발 가이드라인 - 사용 케이스 / 적용 예시

이 문서는 `ai-game-dev-guidelines.md`의 보조 문서다.  
메인 문서가 핵심 요약본이라면, 이 문서는 **A와 B가 실제로 어떻게 다르게 작동하는지**를 짧게 보여준다.

---

## 1. 먼저 기억할 점

A와 B는 둘 다 비슷한 기본 작업 흐름을 따른다.

1. 요구사항 / 개발 계획 정리
2. 작은 단위 구현
3. AI 리뷰 + 인간 리뷰
4. 테스트 준비
5. 실제 테스트
6. 문제 수정 후 반복

차이는 문제를 고치는 기준이다.

- **A:** 문서, 시나리오, 하네스 기준
- **B:** 코드, 구조, 테스트 기준

---

## 2. 같은 문제를 A와 B에서 다르게 푸는 예시

## 예시 1. 결과 UI가 너무 빨리 뜨는 문제

### A에서는
- 요구사항에 “결과 UI는 서버 확정 후에만 노출”을 명시한다.
- 하네스 시나리오에 이벤트 순서 검증을 추가한다.
- AI에게 수정된 요구사항 기준으로 다시 맞추게 한다.
- 다시 실행해서 결과를 확인한다.

즉, **원하는 동작을 더 정확히 정의하고 다시 검증**한다.

### B에서는
- 결과 UI를 띄우는 코드 경로를 찾는다.
- 서버 확정 이벤트와 상태 갱신 순서를 점검한다.
- 필요하면 계약 문서와 상태 전이를 수정한다.
- 코드 패치 후 리뷰와 테스트를 진행한다.

즉, **실제 코드와 구조를 수정해서 해결**한다.

---

## 예시 2. 룸 종료 후 로비 복귀가 가끔 실패하는 문제

### A에서는
- 종료 시나리오를 하네스에 추가한다.
- 요구사항을 더 명확히 적는다.
- mock 세션과 입력값을 조정해 다시 돌린다.

### B에서는
- 세션 종료 처리 코드와 상태 초기화 경로를 본다.
- cleanup 누락, 이벤트 순서, 상태 전이 문제를 수정한다.
- 테스트를 추가하고 다시 검증한다.

---

## 예시 3. 특정 패킷 순서에서 UI가 꼬이는 문제

### A에서는
- packet replay 하네스를 만든다.
- 문제 순서를 재현 가능한 시나리오로 바꾼다.
- AI에게 replay 기준으로 다시 맞추게 한다.

### B에서는
- 패킷 처리 핸들러와 상태 갱신 순서를 분석한다.
- 계약 또는 코드 구조를 수정한다.
- replay / contract test로 편입한다.

---

## 3. A에서 AI를 쓰는 방법

A에서는 AI에게 최종 제품 코드를 요구하기보다,
**빨리 검증할 수 있는 환경**을 만드는 데 집중하는 것이 좋다.

주로 맡길 것:
- 하네스 만들기
- mock / stub 만들기
- 입력 시나리오 정리
- 로그 / 관찰 포인트 추가
- 요구사항 기준으로 다시 생성하기

즉 A에서 AI는 **검증 속도를 올리는 도구**다.

---

## 4. B에서 AI를 쓰는 방법

B에서는 AI를 단순 생성기가 아니라,
**분석 + 계획 + 구현 + 리뷰 + 테스트 보조 도구**로 쓰는 것이 좋다.

주로 맡길 것:
- 기능별 코드베이스 분석
- 코딩 컨벤션 / 구조 규칙 정리
- Gap 분석
- 마이그레이션 전략 문서 초안
- 작은 단위 구현 보조
- 리뷰 포인트 정리
- 테스트 포인트 제안
- 실패 로그 요약

즉 B에서 AI는 **개발 업무를 정리하고 가속하는 도구**에 가깝다.

---

## 5. 실무에서 추천하는 운영 방식

AI에게 모든 걸 한 번에 맡기기보다,
아래처럼 나눠서 쓰는 편이 안정적이다.

1. 분석
2. 계획 문서 작성
3. 구현
4. 리뷰
5. 테스트
6. 재수정

A든 B든 이 방식이 가장 실무적이다.

---

## 6. 마지막 정리

이 문서의 핵심은 간단하다.

- A와 B는 기본 작업 흐름은 비슷하다.
- 차이는 **문제를 무엇을 기준으로 수정하느냐**다.

즉,

- **A는 문서와 하네스 중심**
- **B는 코드와 구조 중심**

이렇게 보면 AI를 훨씬 현실적인 방식으로 운영할 수 있다.
