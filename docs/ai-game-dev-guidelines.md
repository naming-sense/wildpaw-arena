# AI를 활용한 서버/클라이언트 개발 운영 가이드라인

## 1. 핵심 요약

이 문서는 게임 서버/클라이언트 개발에서 AI를 실무에 어떻게 적용할지 정리한 요약본이다.

이 문서에서는 AI 활용 방식을 두 가지로 나눈다.

- **A:** 빠르게 검증하는 방식
- **B:** 메인 프로젝트에 편입하는 방식

핵심은 A와 B가 완전히 다른 방식이 아니라는 점이다.  
둘 다 같은 **기본 작업 흐름**을 따른다.

1. **요구사항 / 개발 계획을 md 파일로 정리한다**
2. **md 파일을 기준으로 작은 단위로 구현한다**
3. **AI 리뷰, 인간 리뷰, 테스트를 진행한다**
4. **수정사항을 정리하고 다시 1번으로 돌아간다**

차이는 문제가 생겼을 때 무엇을 기준으로 수정하느냐다.

- **A는 문서를 수정하면서 검증한다.**
- **B는 코드와 구조를 수정하면서 편입한다.**

---

## 2. A와 B를 나누는 기준

### A를 선택하는 경우
- 지금 필요한 것은 빠른 검증이다.
- 감각, 흐름, 기술 가능성을 먼저 보고 싶다.
- 코드가 나중에 버려지거나 대체돼도 괜찮다.
- 문제가 생기면 요구사항, 개발 계획, 테스트 시나리오를 수정해서 다시 확인할 생각이다.

### B를 선택하는 경우
- 메인 프로젝트에 넣어야 한다.
- 다른 개발자와 협업해야 한다.
- 코드 리뷰, 테스트, 유지보수가 필요하다.
- 문제가 생기면 실제 코드와 구조를 분석해서 고쳐야 한다.

한 줄로 정리하면:

- **A는 검증용**
- **B는 편입용**

---

## 3. 기본 작업 흐름

A와 B 모두 아래 흐름을 기본으로 한다.

### 1) 요구사항 / 개발 계획을 md 파일로 정리
최소한 아래는 먼저 적어둔다.

- 목표
- 범위
- 제외 범위
- 완료 기준
- 테스트 포인트
- 이번 단계에서 다룰 작업 단위

### 2) md 파일을 기준으로 작은 단위 구현
AI에게 큰 기능 전체를 한 번에 맡기지 않고, 문서 기준으로 작은 단위 구현을 진행한다.

### 3) AI 리뷰, 인간 리뷰, 테스트 진행
구현이 끝나면 바로 다음 작업으로 넘기지 않고 다음을 확인한다.

- 요구사항 누락
- 구조 문제
- 상태 전이 문제
- 예외 상황 누락
- 테스트 부족

### 4) 수정사항 정리 후 다시 1번으로
문제가 나오면 수정사항을 정리하고, 문서를 갱신한 다음 다시 진행한다.

이 흐름을 반복하면서 결과물을 다듬는다.

---

## 4. A에서 일하는 방식

A는 빠르게 검증하는 방식이다.  
중요한 것은 코드 자체보다 **요구사항과 결과가 맞는지 빠르게 확인하는 것**이다.

### A에서 1번 문서에 주로 들어가는 것
- 무엇을 검증할 것인지
- 어떤 결과가 나오면 성공인지
- 어떤 테스트 시나리오로 확인할지
- 이번 단계에서 제외할 것은 무엇인지

### A에서 문제가 생기면 먼저 보는 것
1. 요구사항이 모호한가?
2. 개발 계획 범위가 너무 넓은가?
3. 테스트 시나리오가 부족한가?
4. 결과 판단 기준이 불명확한가?

### A에서 주로 하는 수정
- 요구사항 문구 수정
- 개발 계획 범위 재정리
- 테스트 시나리오 추가
- 테스트용 데이터나 입력값 조정
- 다시 구현하고 다시 확인

즉 A는 **문서를 수정하면서 검증을 반복하는 방식**이다.

### A를 계속 끌면 안 되는 순간
- 원인 분석에 코드 이해가 필수일 때
- 다른 개발자가 이어받아야 할 때
- 메인 프로젝트 편입을 본격적으로 검토할 때

이때는 B로 넘어가는 게 맞다.

---

## 5. B에서 일하는 방식

B는 메인 프로젝트에 편입하는 방식이다.  
기본 작업 흐름은 같지만, 1번 문서를 만드는 단계가 훨씬 중요하고 무겁다.

B에서는 보통 1번 문서를 정리하기 전에 아래 작업이 먼저 필요하다.

1. **기존 프로젝트 기능별 코드베이스 분석**
2. **코딩 컨벤션 / 구조 규칙 파악**
3. **현재 프로젝트와 대상 모듈 간 차이 정리**
4. **마이그레이션 전략 / 요구사항 / 개발 계획 문서화**

즉 B에서는 문서를 그냥 작성하는 것이 아니라,
**기존 프로젝트를 분석한 결과를 바탕으로 문서를 만드는 것**이 핵심이다.

### B에서 문제가 생기면 먼저 보는 것
1. 어떤 코드 경로에서 문제가 나는가?
2. 구조, 책임 분리, 상태 전이가 맞는가?
3. 컨벤션이나 계약을 어기고 있지 않은가?
4. 테스트가 부족한가?

### B에서 주로 하는 수정
- 실제 코드 패치
- 구조 조정
- 계약 문서 수정
- 테스트 추가 / 수정
- 리뷰 후 재검증

즉 B는 **기존 프로젝트 분석 + 문서화 + 코드 수정**이 함께 가는 방식이다.

---

## 6. AI를 실무에 적용할 때의 원칙

### 1) AI를 한 번에 다 쓰지 않는다
다음처럼 역할을 나눠서 쓰는 편이 안정적이다.

- 분석
- 문서 작성
- 구현
- 리뷰
- 테스트
- 재수정

### 2) 항상 작은 단위로 자른다
AI는 큰 범위를 한 번에 맡기면 품질이 흔들리기 쉽다.

### 3) AI를 코드 생성기로만 보지 않는다
AI는 다음 역할에도 유용하다.

- 코드베이스 분석
- 컨벤션 정리
- 문서 초안 작성
- 리뷰 보조
- 테스트 케이스 제안
- 실패 로그 요약

### 4) A와 B를 섞지 않는다
- A는 검증에 집중
- B는 편입과 유지보수에 집중

이 구분이 흐려지면 속도도 품질도 같이 떨어진다.

---

## 7. 결론

AI를 활용한 서버/클라이언트 개발에서 중요한 것은 복잡한 용어가 아니라,
**문서를 기준으로 반복 작업을 돌리되 A와 B가 무엇을 수정하는지 구분하는 것**이다.

- **A는 문서를 수정하면서 검증한다.**
- **B는 기존 프로젝트 분석과 문서화를 바탕으로 코드와 구조를 수정한다.**

이 기준만 분명해도 AI를 훨씬 현실적으로 운영할 수 있다.
