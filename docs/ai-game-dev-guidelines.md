# AI를 활용한 서버/클라이언트 개발 운영 가이드라인

## 1. 문서 목적

이 문서는 게임 **서버/클라이언트 개발 업무에 AI를 실제로 적용하는 방법**을 정리한 운영 가이드다.

이번 버전에서는 A와 B를 다음처럼 정의한다.

- **Case A:** 하네스 엔지니어링(Harness Engineering) 기반 프로토타이핑 / MVP 검증
- **Case B:** 협업과 메인 프로젝트 마이그레이션을 위한 구조화된 개발 프로세스

핵심은 단순하다.

- A는 **빠르게 검증하기 위한 단계**다.
- B는 **팀이 유지할 수 있는 형태로 편입하기 위한 단계**다.

즉, A와 B의 차이는 “코드를 보느냐 안 보느냐”보다,  
**지금 무엇을 검증하려는지 / 누구와 함께 유지할 것인지 / 메인 프로젝트에 넣을 것인지**에 있다.

---

## 2. A / B 구분 기준

### 2.1 Case A

Case A는 AI를 활용해 **빠르게 실험 가능한 모듈과 검증용 하네스까지 함께 만드는 방식**이다.

여기서 중요한 것은 “완성 코드”가 아니라 **검증 가능한 실행 환경**이다.

예를 들어 A에서는 다음을 만든다.

- 클라이언트 전투 감각을 검증하는 입력/애니메이션 하네스
- 서버 이벤트 순서를 검증하는 세션 시뮬레이션 하네스
- 패킷 흐름을 재생하는 replay 하네스
- 고정된 시나리오를 반복 실행하는 테스트 더미 환경
- 실제 메인 프로젝트 전체를 띄우지 않고 특정 모듈만 확인할 수 있는 실행기

즉 A는 “AI가 코드를 만들었다”가 핵심이 아니라,  
**AI가 빠르게 검증 가능한 환경까지 함께 만들게 해서 시행착오 비용을 줄이는 것**이 핵심이다.

### 2.2 Case B

Case B는 AI를 활용하되 결과물을 **협업 가능한 코드와 문서 자산으로 변환**하는 방식이다.

여기서는 다음이 중요하다.

- 기존 프로젝트 구조 이해
- 기능별 코드베이스 분석
- 코딩 컨벤션 파악
- A에서 만든 모듈의 마이그레이션 전략 수립
- 문서 기반 구현
- 코드 리뷰
- 테스트 프레임워크 적용
- 실제 테스트
- 이슈 발생 시 반복 이터레이션

즉 B는 “AI로 빨리 구현”이 아니라,  
**AI를 분석자 + 설계 보조 + 구현 보조 + 리뷰 보조 + 테스트 보조로 배치하는 운영 프로세스**다.

### 2.3 한 줄 기준

- **새로운 감각, 구조, 흐름을 빨리 검증해야 하면 A**
- **이미 검증된 것을 팀 코드베이스로 옮겨야 하면 B**

### 2.4 전환 기준

다음 중 하나라도 생기면 A에서 B로 넘어간다.

- 프로토타입이 실제 제품 기능으로 채택되었다.
- 다른 개발자가 이어받아야 한다.
- 메인 프로젝트 레포에 넣어야 한다.
- 서버/클라이언트 계약을 고정해야 한다.
- 유지보수와 테스트 체계가 필요해졌다.

## 3. Case A - 하네스 엔지니어링 기반 프로토타이핑 / MVP

## 3.1 A를 하네스 엔지니어링 관점으로 보는 이유

보통 프로토타입 단계에서는 AI에게 바로 기능 구현을 맡기기 쉽다.  
그런데 이렇게 하면 결과가 빨리 나오더라도 다음 문제가 생긴다.

- 메인 프로젝트와 너무 일찍 엮인다.
- 실험 범위가 불분명해진다.
- 무엇을 검증한 것인지 기록이 남지 않는다.
- 나중에 버리거나 옮기기 어려워진다.

그래서 A는 **기능 자체**보다 먼저, 그 기능을 빠르게 검증할 수 있는 **하네스(harness)** 를 같이 설계하는 방식으로 보는 것이 좋다.

여기서 하네스란,

> 특정 모듈이나 흐름을 메인 프로젝트 전체와 분리해 반복 실행하고 관찰할 수 있게 만드는 검증용 실행 환경

을 의미한다.

즉 A의 목적은 “AI가 만든 코드”가 아니라,
**AI가 만든 코드가 빠르게 실험되고 반복 검증될 수 있는 구조를 만드는 것**이다.

---

## 3.2 A에서 하네스가 가져야 할 요소

A에서 만드는 하네스는 보통 아래 요소를 가진다.

### 1) 검증 대상 경계
무엇을 검증할지 먼저 자른다.

예:
- 이동/대시 감각만 검증
- 룸 상태 전이만 검증
- 패킷 송수신 순서만 검증
- UI 결과 화면 진입 흐름만 검증

### 2) 입력 제어 장치
반복 가능한 입력이 있어야 한다.

예:
- 키 입력 시퀀스
- 더미 패킷 목록
- 세션 이벤트 재생 스크립트
- mock 데이터

### 3) 관찰 포인트
결과를 눈으로 보거나 로그로 확인할 수 있어야 한다.

예:
- 프레임별 상태 로그
- 이벤트 순서 출력
- HUD 반응 캡처
- 시뮬레이션 결과 요약

### 4) 빠른 리셋 가능성
한 번 돌리고 끝나는 게 아니라 반복 실행 가능해야 한다.

### 5) 메인 프로젝트와의 느슨한 연결
되도록 직접 메인 시스템을 오염시키지 않는다.

---

## 3.3 Case A 작업 순서

### A-1. 검증 질문 정의
먼저 “이번 사이클에서 무엇을 검증할지”를 적는다.

예:
- 이 전투 루프가 재미있는가?
- 룸 시작 → 플레이 → 종료 흐름이 기술적으로 성립하는가?
- 이 패킷 구조로 클라이언트 UI까지 연결 가능한가?

### A-2. 하네스 경계 설계
어디까지를 검증 대상으로 보고, 어디부터는 mock으로 처리할지 정한다.

예:
- 실제 네트워크 대신 로컬 이벤트 큐 사용
- 실제 인증 대신 더미 세션 사용
- 실제 매치메이킹 대신 고정 룸 시나리오 사용

### A-3. AI에 하네스 포함 요청
AI에게 기능만 시키지 말고 다음까지 같이 요청한다.

- 실행 방법
- mock/stub 데이터
- 시나리오 입력
- 결과 관찰 포인트
- 빠른 재실행 방법

즉, “기능 구현”이 아니라 **기능 + 검증 환경**을 요청해야 한다.

### A-4. AI 구현 및 반복 실행
AI가 하네스와 기능을 만들면 사람은 빠르게 반복 실행하며 다음을 본다.

- 감각이 맞는가
- 이벤트 흐름이 자연스러운가
- 병목이나 구조 문제가 보이는가
- 예상치 못한 실패 케이스가 있는가

### A-5. 결과 기록
A에서는 다음을 반드시 남긴다.

- 무엇을 검증했는가
- 어떤 하네스를 만들었는가
- 어떤 시나리오를 돌렸는가
- 어떤 결과가 나왔는가
- 무엇이 임시 구현인가
- 메인 프로젝트 편입 시 주의점은 무엇인가

### A-6. B 전환 여부 결정
A 결과가 유효하면 B로 간다.  
유효하지 않으면 폐기하거나 다른 하네스로 재실험한다.

---

## 3.4 A에서 AI를 어떻게 쓰는가

A에서 AI는 주로 아래 역할을 한다.

- 하네스 초안 설계자
- mock/stub 생성기
- 프로토타입 구현자
- 시나리오 러너 작성 보조
- 로그/결과 요약 보조

좋은 A 요청은 보통 이렇게 생긴다.

```md
목표:
- 대시 전투 감각을 검증한다.

요청:
- 플레이 가능한 최소 모듈을 만든다.
- 입력 시퀀스를 반복 실행할 수 있는 하네스를 함께 만든다.
- 선딜, 무적 프레임, 쿨다운 값을 쉽게 바꿀 수 있게 한다.
- 실행 결과를 로그와 화면에서 확인할 수 있게 한다.

제외:
- 최종 상태 관리 구조
- 실제 멀티플레이 동기화
- 메인 프로젝트 전체 통합
```

핵심은 **AI에게 최종 제품 코드를 요구하지 말고, 검증 가능한 실험 단위를 요구하는 것**이다.

---

## 3.5 A의 산출물

A 단계 종료 시 남겨야 할 최소 산출물은 다음이다.

- Harness Summary
- 검증 질문 목록
- 실행 시나리오 목록
- 결과 요약
- Known Limitations
- Migration Notes

여기서 특히 중요한 것은 **Migration Notes**다.  
A에서 만든 것을 B로 옮길 때,
“무엇은 살리고 무엇은 버릴지”의 출발점이 된다.

---

## 4. Case B - 협업 / 메인 프로젝트 마이그레이션 운영 프로세스

## 4.1 B의 목적

B의 목적은 A에서 검증된 아이디어나 모듈을,  
**기존 프로젝트 구조와 팀의 작업 방식에 맞는 형태로 편입**하는 것이다.

여기서 중요한 것은 구현 그 자체보다 **업무 순서와 산출물의 품질**이다.

즉 B는 다음 순서로 운영하는 것이 좋다.

1. 기존 프로젝트를 이해한다.
2. 프로젝트 규칙을 파악한다.
3. A 결과물과 현재 코드베이스의 간극을 정리한다.
4. 마이그레이션 요구사항과 개발 계획을 문서화한다.
5. 문서를 기준으로 구현한다.
6. 구현 결과를 리뷰한다.
7. 테스트 프레임워크와 실제 테스트를 적용한다.
8. 이슈가 생기면 반복 이터레이션한다.
9. 협업 가능한 상태로 정리하고 넘긴다.

---

## 4.2 B 작업 순서 상세

### B-1. 기존 프로젝트 기능별 코드베이스 분석

가장 먼저 해야 할 일은 **현재 프로젝트가 이미 어떻게 동작하는지 기능 단위로 파악하는 것**이다.

#### 분석 대상 예시
- 로그인/세션 흐름
- 로비 흐름
- 룸 생성/입장/퇴장
- 매치메이킹
- 인게임 상태 관리
- UI 갱신 구조
- 서버 이벤트 처리 구조
- 패킷/메시지 스키마
- 로그/에러 처리 패턴

#### AI 활용 방식
AI에게 관련 파일 묶음을 주고 다음을 정리하게 한다.

- 기능별 책임 분리
- 주요 진입점(entrypoint)
- 호출 흐름
- 관련 파일 목록
- 의존 관계
- 수정 시 영향 범위
- 위험 포인트

#### 이 단계의 산출물
- 기능별 코드베이스 분석 문서
- 관련 파일 맵
- 의존성 요약
- 영향 범위 목록

#### 실무 팁
AI에게 코드 전체를 한 번에 던지기보다,
**기능별로 나눠서 분석하게 하는 것이 훨씬 정확하다.**

---

### B-2. 기존 프로젝트 코딩 컨벤션 및 구조 규칙 파악

기능 분석 다음에는 **이 프로젝트가 어떤 스타일로 코드를 쓰는지** 파악해야 한다.

여기서 보는 것은 단순 포매팅만이 아니다.

#### 파악 대상
- 네이밍 규칙
- 폴더 구조
- 클래스/모듈 책임 분리 방식
- 상태 관리 방식
- 에러 처리 방식
- 비동기 처리 방식
- 로그 스타일
- 주석 스타일
- 테스트 작성 방식
- PR 단위 관례

#### AI 활용 방식
AI에게 다음 질문을 시킨다.

- 이 프로젝트에서 공통적으로 반복되는 패턴은 무엇인가?
- 새 모듈이 들어오면 어디에 위치해야 하는가?
- 어떤 스타일은 지켜야 하고 어떤 스타일은 피해야 하는가?
- 암묵적인 팀 규칙처럼 보이는 것은 무엇인가?

#### 산출물
- Coding Convention Snapshot
- Architecture Rule Summary
- Do / Don’t 목록

이 단계를 건너뛰면,
AI는 “좋아 보이는 코드”를 만들 수는 있어도 **이 프로젝트에 맞는 코드**를 만들 가능성은 낮아진다.

---

### B-3. A 모듈과 현재 프로젝트 간 Gap 분석

이제 A에서 만든 모듈을 그대로 넣을 수 있는지 보지 말고,
**현재 프로젝트와 어떤 차이가 있는지**를 먼저 정리한다.

#### Gap 분석 항목
- 진입점 차이
- 상태 관리 방식 차이
- 데이터 구조 차이
- 서버/클라이언트 계약 차이
- 로깅/디버깅 체계 차이
- 에러 처리 정책 차이
- 테스트 가능성 차이
- 성능/메모리 제약 차이

#### AI 활용 방식
AI에게 다음 형식으로 비교하게 한다.

- A 모듈의 현재 구조
- 메인 프로젝트의 요구 구조
- 그대로 재사용 가능한 부분
- 재작성해야 하는 부분
- 인터페이스만 유지하고 내부는 다시 짜야 하는 부분

#### 산출물
- Gap Analysis 문서
- 재사용 / 재작성 분류표
- 리스크 목록

---

### B-4. 마이그레이션 전략의 요구사항 및 개발 계획 문서화

이 단계가 B의 핵심이다.

A에서 만든 모듈을 프로젝트에 넣기 전에,
**무엇을 어떤 방식으로 옮길지 문서로 먼저 고정**해야 한다.

#### 문서에 들어가야 할 항목
- 목표 기능
- 현재 프로젝트 내 편입 위치
- 유지할 로직
- 버릴 임시 구현
- 새로 설계할 인터페이스
- 서버/클라이언트 계약 정리
- 구현 단계 순서
- 파일 단위 변경 계획
- 테스트 전략
- 리스크 및 롤백 포인트

#### AI 활용 방식
AI에게 다음을 시킨다.

- 마이그레이션 전략 초안 작성
- 단계별 개발 계획 작성
- 구현 순서 제안
- 의존성 순서 정리
- 리뷰 포인트 정리
- 테스트 포인트 정리

#### 좋은 운영 방식
이 문서는 단순 메모가 아니라,
**이후 구현·리뷰·테스트의 기준 문서**가 되어야 한다.

즉 B에서는 “바로 구현”보다 **문서 먼저**가 더 중요하다.

---

### B-5. 문서 기반 구현

이제 문서를 기준으로 실제 구현에 들어간다.

중요한 점은 A처럼 한 번에 크게 구현하지 말고,
**리뷰 가능한 단위로 쪼개는 것**이다.

#### 권장 분할 방식
- 계약 정리 변경
- 데이터 모델 변경
- 서버 핸들러 변경
- 클라이언트 수신 처리 변경
- UI 연결 변경
- 테스트 코드 추가

#### AI 활용 방식
AI에게 구현을 맡길 때는 항상 다음을 같이 요구한다.

- 변경 이유
- 변경 파일 목록
- 영향 범위
- 남아 있는 TODO
- 테스트 포인트

#### 실무 팁
AI에게 “전부 구현해줘”보다,
“문서의 2단계까지만 구현해줘”가 훨씬 안전하다.

---

### B-6. 구현 내용 코드 검토

구현 후에는 바로 머지하지 말고,
**AI 리뷰 + 인간 리뷰**를 분리해 운영하는 것이 좋다.

#### 1차: AI 자체 리뷰 / 보조 리뷰
AI에게 다음을 점검하게 한다.

- 컨벤션 위반
- 구조 위반
- 책임 과다 집중
- 예외 처리 누락
- 테스트 누락
- 사이드이펙트 가능성
- 로그 부족 지점

#### 2차: 인간 리뷰
사람은 특히 아래를 본다.

- 실제 팀 구조와 맞는가
- 다른 개발자가 이어받을 수 있는가
- 장기 유지보수에 문제 없는가
- 계약 변경이 다른 기능을 깨지 않는가

#### 산출물
- Code Review Notes
- 수정 필요 항목 목록
- 승인 / 수정 후 승인 / 재설계 필요

---

### B-7. 테스팅 프레임워크 적용

형진님이 말한 흐름에서 중요한 포인트 중 하나가 바로 이 부분이다.

B에서는 구현 후 테스트만 하는 것이 아니라,
필요하면 **테스트 프레임워크 자체를 먼저 붙이는 작업**도 별도 단계로 봐야 한다.

#### 왜 필요한가
A 단계에서는 하네스로 빠르게 검증하지만,
B에서는 반복 가능한 팀 테스트 체계가 필요하다.

#### 적용 대상 예시
- 서버 단위 테스트 프레임워크
- 패킷/이벤트 contract test
- 클라이언트 상태 전이 테스트
- 세션 흐름 smoke test
- replay 기반 regression test

#### AI 활용 방식
AI에게 다음을 요청할 수 있다.

- 현재 프로젝트에 맞는 테스트 진입점 제안
- 최소 테스트 구조 추가
- fixture / mock / test helper 생성
- CI에서 돌릴 수 있는 테스트 세트 초안 작성

#### 산출물
- Test Framework Setup Notes
- Test Helper / Fixture 구조
- 실행 명령 정리

---

### B-8. 실제 테스트 수행

프레임워크를 붙였다고 끝이 아니다.  
실제 테스트를 돌리고 결과를 읽어야 한다.

#### 테스트 종류
- 자동 테스트
- 통합 테스트
- 서버/클라이언트 연결 테스트
- 수동 플레이 테스트
- 예외 흐름 테스트
- 성능/안정성 확인

#### AI 활용 방식
AI는 여기서 다음 보조 역할을 할 수 있다.

- 테스트 케이스 제안
- 실패 로그 요약
- 재현 절차 문서화
- 실패 원인 가설 정리
- 회귀 가능성 높은 지점 표시

#### 산출물
- Test Result Report
- 실패 케이스 목록
- 재현 절차
- 우선순위 분류

---

### B-9. 문제 발생 시 반복 이터레이션

실무에서는 이 단계가 빠질 수 없다.

B는 선형 파이프라인이 아니라,
**문제 유형에 따라 앞 단계로 되돌아가는 반복 구조**로 운영해야 한다.

#### 대표적인 되돌림 규칙
- 컨벤션 문제면 → B-2로
- 구조 문제면 → B-3 또는 B-4로
- 구현 문제면 → B-5로
- 테스트 누락이면 → B-7로
- 실제 실패 재현이면 → B-8 결과를 바탕으로 B-4~B-5 재정의

#### AI 활용 방식
AI는 반복 이터레이션에서 특히 강하다.

- 실패 로그 요약
- 원인 후보 목록화
- 수정 우선순위 정리
- 재테스트 범위 추천
- 회귀 의심 영역 다시 점검

즉, AI를 “코드 생성기”로만 보지 말고,
**반복 사이클을 짧게 만드는 분석 보조자**로 쓰는 것이 좋다.

---

### B-10. 협업 가능한 상태로 정리 및 인수인계

최종적으로는 구현보다 **넘길 수 있는 상태**가 중요하다.

#### 남겨야 할 것
- 무엇을 왜 바꿨는가
- 어떤 문서를 기준으로 구현했는가
- 어떤 테스트를 추가했는가
- 아직 남은 리스크는 무엇인가
- 다음 개발자가 어디부터 보면 되는가

#### 최종 산출물
- Migration Plan 최종본
- 구현 요약
- 리뷰 요약
- 테스트 결과
- Known Issues / Follow-up

## 4.3 B를 실제 업무에 적용하는 추천 운영 방식

B에서 AI를 잘 쓰려면, 한 번의 대화에 모든 역할을 몰아넣지 않는 것이 중요하다.

다음처럼 **역할을 분리해서 쓰는 방식**이 실무에서 더 잘 맞는다.

### 1) 분석 턴
AI에게 코드베이스 분석만 맡긴다.

### 2) 규칙 추출 턴
AI에게 컨벤션과 구조 규칙만 뽑게 한다.

### 3) 계획 턴
AI에게 마이그레이션 전략 문서만 쓰게 한다.

### 4) 구현 턴
문서를 기준으로 작은 단위 구현만 시킨다.

### 5) 리뷰 턴
구현한 내용을 리뷰하게 한다.

### 6) 테스트 턴
테스트 구조와 테스트 케이스를 정리하게 한다.

### 7) 이터레이션 턴
문제 발생 시 실패 원인과 다음 액션만 정리하게 한다.

이렇게 역할을 분리하면,
AI가 한 번에 너무 많은 판단을 하다가 품질이 흐트러지는 문제를 줄일 수 있다.

## 6. A와 B를 함께 운영하는 실무 해석

정리하면,

### A는 이렇게 쓰면 된다
- AI에게 기능과 함께 **검증용 하네스**를 만들게 한다.
- 메인 프로젝트에 넣기 전에 실험 가능한 실행 환경을 확보한다.
- 빠르게 돌려보고, 실패 비용을 낮춘다.
- 결과와 Migration Notes를 남긴다.

### B는 이렇게 쓰면 된다
- AI에게 먼저 현재 프로젝트를 분석하게 한다.
- 코드베이스 구조와 컨벤션을 파악하게 한다.
- A 결과물을 바로 넣지 말고, 마이그레이션 문서를 먼저 쓰게 한다.
- 문서 기준으로 작은 단위 구현을 진행한다.
- 구현 후 리뷰, 테스트 프레임워크, 실제 테스트를 붙인다.
- 문제 발생 시 반복 이터레이션을 운영한다.

즉,

- A에서 AI는 **실험 가속기**
- B에서 AI는 **분석자 + 계획 보조 + 구현 보조 + 리뷰 보조 + 테스트 보조**

로 쓰는 것이 가장 실무적이다.

---

## 7. 최종 결론

서버/클라이언트 개발에서 AI를 제대로 쓰려면,
프로토타입 단계와 메인 프로젝트 편입 단계를 같은 방식으로 다루면 안 된다.

- **A는 하네스 엔지니어링 기반으로 빠르게 검증하는 단계**
- **B는 기존 코드베이스 분석부터 마이그레이션, 구현, 리뷰, 테스트, 반복 이터레이션까지 포함하는 운영 단계**

이 기준으로 보면 AI는 단순 코더가 아니다.

- A에서는 **빠른 실험 환경을 만드는 도구**이고,
- B에서는 **실제 개발 파이프라인을 가속하는 협업 도구**다.

이렇게 나누어 써야, AI를 공격적으로 활용해도 메인 프로젝트 품질과 협업 생산성을 같이 가져갈 수 있다.
