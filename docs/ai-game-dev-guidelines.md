# AI를 활용한 게임 제작 가이드라인

## 1. 문서 목적

이 문서는 **AI를 활용한 게임 제작 작업을 운영 가능한 프로세스로 표준화**하기 위한 가이드라인이다.
특히 다음 두 가지 케이스를 분리해 정의한다.

- **A. 블랙박스형 결과 검수 프로세스**  
  AI가 작성한 코드를 사람이 직접 리뷰하지 않고, **요구사항과 실행 결과물만 검수**하는 방식
- **B. 화이트박스형 코드 리뷰 프로세스**  
  AI가 작성한 코드를 사람이 **리뷰하고 수정 방향을 통제**하면서 진행하는 방식

핵심 목적은 다음과 같다.

1. 작업 성격에 따라 AI 활용 방식을 명확히 구분한다.
2. 품질, 속도, 유지보수성 사이의 균형을 잡는다.
3. 게임 개발 특성상 중요한 성능, 안정성, 반복 수정 비용을 관리한다.
4. 팀이 동일한 기준으로 작업을 발주하고 검수할 수 있게 한다.

---

## 2. 적용 범위

이 가이드는 다음 유형의 작업에 적용한다.

- 게임플레이 기능 구현
- UI/UX 기능 구현
- 툴 제작 및 에디터 스크립트
- 전투/퀘스트/이벤트/레벨 스크립트
- 데이터 가공 및 밸런싱 보조 도구
- 빌드/테스트 자동화 스크립트
- 프로토타입 및 실험 기능

적용 제외 또는 별도 심의가 필요한 영역:

- 결제/과금
- 계정/인증
- 보안 기능
- 안티치트
- 서버 권한 로직
- 저장 데이터 마이그레이션
- 라이브 서비스 핵심 장애 대응 코드
- 법적/라이선스 검토가 필요한 외부 코드 혼합

이 영역들은 원칙적으로 **Case B 또는 사람 주도 구현**을 기본으로 한다.

## 3. 기본 원칙

### 3.1 사람 책임 원칙
AI는 생산 도구이며, 최종 책임은 항상 사람에게 있다.  
출시 승인, 머지 승인, 라이브 반영 결정은 반드시 사람이 내린다.

### 3.2 요구사항 선명화 원칙
AI의 품질보다 먼저 중요한 것은 **작업 지시의 품질**이다.  
불명확한 요구사항은 높은 확률로 불필요한 수정 비용을 만든다.

### 3.3 작은 단위 분할 원칙
큰 기능을 한 번에 맡기지 말고, **작업 단위를 작게 분해**한다.

- 한 기능
- 한 화면
- 한 시스템의 한 서브모듈
- 한 종류의 버그 수정

### 3.4 테스트 우선 원칙
AI에게 구현을 맡기기 전에 **완료 기준과 테스트 시나리오**를 먼저 정의한다.

### 3.5 위험도 기반 선택 원칙
속도만 보고 Case A를 선택하지 않는다.  
작업의 장기 유지 가능성, 핵심도, 장애 비용을 기준으로 케이스를 선택한다.

### 3.6 기록 원칙
모든 AI 작업에는 최소한 다음 기록이 남아야 한다.

- 작업 목적
- 입력 요구사항
- 사용한 제약 조건
- 검수 결과
- 채택/반려 이유
- 후속 액션

## 4. 케이스 정의

## 4.1 Case A: 블랙박스형 결과 검수 프로세스

### 정의
AI가 만든 내부 코드는 세밀하게 검토하지 않고, 다음만 기준으로 판단한다.

- 요구사항 충족 여부
- 실제 플레이/실행 결과
- 테스트 통과 여부
- 성능 및 안정성 지표

### 적합한 상황

- 빠른 프로토타이핑
- 폐기 가능성이 높은 실험 기능
- 임시 이벤트/연출 스크립트
- 내부 툴
- 코드보다 결과 동작이 중요한 작업
- 짧은 생명주기의 구현물

### 부적합한 상황

- 장기적으로 계속 확장될 핵심 시스템
- 다른 시스템과 깊게 엮이는 공용 모듈
- 성능/메모리 민감 기능
- 멀티플레이 동기화
- 저장/불러오기 핵심 로직
- 결제, 계정, 권한 처리

### 핵심 특징

- **속도 우선**
- **결과 중심 검수**
- **코드 품질보다 출력 품질에 집중**
- 유지보수성에 한계가 생기면 Case B로 승격

---

## 4.2 Case B: 화이트박스형 코드 리뷰 프로세스

### 정의
AI가 만든 코드와 변경사항을 사람이 직접 읽고, 구조/품질/위험을 검토하며 진행하는 방식이다.

### 적합한 상황

- 운영 제품 코드
- 공용 프레임워크/베이스 시스템
- 성능 민감 기능
- 반복 확장이 예상되는 기능
- 협업 코드베이스
- 버그 재현과 원인 분석이 중요한 이슈

### 핵심 특징

- **품질과 통제 우선**
- **작은 변경 단위**
- **사람 리뷰 필수**
- 구조적 일관성과 유지보수성 확보 가능

---

## 5. 케이스 선택 기준

| 기준 | Case A 권장 | Case B 권장 |
|---|---|---|
| 목적 | 빠른 검증, 실험, 프로토타입 | 제품 코드, 장기 운영 |
| 수명 | 짧음 | 김 |
| 코드 재사용 가능성 | 낮음 | 높음 |
| 시스템 결합도 | 낮음 | 높음 |
| 장애 비용 | 낮음 | 높음 |
| 성능 민감도 | 낮음 | 높음 |
| 보안/데이터 위험 | 낮음 | 높음 |
| 팀 협업 영향도 | 낮음 | 높음 |
| 인간 개입 방식 | 결과물 검수 | 코드 리뷰 + 결과물 검수 |

### 빠른 결정 규칙
다음 중 하나라도 해당하면 **Case B 우선**:

- 핵심 게임 루프에 직접 연결된다.
- 다른 기능들이 이 코드를 기반으로 확장될 예정이다.
- 디버깅 시 코드 이해가 필수다.
- 성능 저하가 플레이 체감에 직접 영향을 준다.
- 저장 데이터, 네트워크, 보안, 결제와 연결된다.
- 출시 후 장기간 유지보수해야 한다.

# 7. Case A 파이프라인 - 블랙박스형 결과 검수 프로세스

## 개요 흐름

**작업 선정 → 요구사항 패키징 → 검수 기준 정의 → AI 구현 → 실행 결과 검수 → 수정 요청 반복 → 채택 또는 폐기**

---

## A-1. 작업 선정 및 격리 여부 판단

### 목표
해당 작업이 정말 Case A에 적합한지 먼저 판단한다.

### 수행 내용

- 작업의 수명 주기를 판단한다.
- 공용 시스템 의존도를 점검한다.
- 실패 시 피해 범위를 점검한다.
- 가능하면 별도 브랜치, 샌드박스, 실험 폴더에서 진행한다.

### 산출물

- Case A 채택 여부
- 격리 범위 정의
- 실패 시 폐기 가능 여부 확인

### 승인 기준
다음 조건을 만족해야 Case A로 간다.

- 코드 내부를 몰라도 결과 검수가 가능하다.
- 실패해도 시스템 전체에 큰 손상이 없다.
- 버려도 되는 구현이거나 재작성 비용이 감당 가능하다.

---

## A-2. 요구사항 패키징

### 목표
AI가 결과 중심으로 구현할 수 있도록 입력을 명확하게 정리한다.

### 필수 입력 항목

- 기능 목적
- 플레이어가 보게 될 최종 동작
- 입력/출력 조건
- UI/UX 요구사항
- 예외 케이스
- 플랫폼 제약
- 엔진/프레임워크 버전
- 수정 가능한 파일 범위
- 금지 사항

### 작성 원칙

- 내부 구현 방식을 지시하기보다 **기대 결과**를 명확히 쓴다.
- 수치화 가능한 기준을 넣는다.
- 애매한 표현을 줄인다.

### 예시

- 잘못된 요구: “전투를 더 재밌게 만들어줘”
- 좋은 요구: “플레이어가 스페이스 입력 시 0.2초 선딜 후 대시하며, 1.5초 쿨다운이 적용되고, 대시 중 피격 판정은 유지된다.”

---

## A-3. 완료 기준 및 테스트 시나리오 정의

### 목표
코드를 보지 않아도 통과/실패를 판정할 수 있게 한다.

### 필수 기준

- 기능 테스트
- 예외 테스트
- 회귀 테스트
- 성능 기준
- 빌드 가능 여부

### 게임 개발 기준 예시

- 60 FPS 기준 프레임 드랍 없음
- 특정 상황에서 애니메이션 끊김 없음
- UI 해상도 대응 정상
- 저장/로드 후 상태 유지
- 기존 입력 체계와 충돌 없음

### 체크 포인트
Case A에서는 **테스트 시나리오가 코드 리뷰를 대신한다.**  
따라서 테스트 설계가 빈약하면 전체 프로세스가 무너진다.

---

## A-4. AI 구현 요청

### 목표
AI가 불필요한 변경 없이 요구사항에 맞는 결과를 만들게 한다.

### 요청 템플릿

1. 작업 목적
2. 기대 동작
3. 수정 허용 범위
4. 금지 범위
5. 완료 기준
6. 테스트 시나리오
7. 반환 형식

### 운영 원칙

- 한 번에 한 작업만 맡긴다.
- 큰 기능은 단계별로 쪼갠다.
- “이 구조가 맞는지”보다 “이 결과를 만족하는지”를 본다.
- 필요 시 동일 요구사항으로 여러 버전을 받아 비교한다.

---

## A-5. 결과물 검수

### 목표
코드를 읽지 않고 실행 결과만으로 채택 여부를 판단한다.

### 검수 항목

- 요구사항 충족 여부
- 플레이 체감
- UX 자연스러움
- 버그 발생 여부
- 성능 기준 충족 여부
- 기존 시스템과 충돌 여부

### 검수 방법

- 실제 플레이 테스트
- 체크리스트 기반 확인
- QA 시나리오 실행
- 영상 캡처/로그 수집
- 비교 빌드 검증

### 권장 규칙

- 검수자는 가능하면 구현 요청자와 분리한다.
- “대충 맞는 것 같음”은 통과 기준이 아니다.
- 결과 검수에서 애매함이 남으면 반려한다.

---

## A-6. 수정 요청 반복

### 목표
코드 수정 지시가 아니라 **동작 수정 지시**로 반복 개선한다.

### 수정 요청 방식

- 어떤 시나리오에서 실패했는지 명시
- 기대한 결과와 실제 결과를 비교
- 재현 절차를 적음
- 우선순위를 붙임

### 예시

- “코드 구조를 정리해줘” 대신
- “연속 대시 입력 시 쿨다운이 무시되는 문제를 수정해줘. 3회 반복 입력 테스트에서 항상 1.5초 쿨다운이 유지되어야 함.”

### 중요 원칙
Case A에서는 사람이 내부 코드를 만지기 시작하는 순간, 프로세스가 모호해진다.  
사람이 직접 내부 구현을 이해하고 수정해야 한다면 **Case B로 승격**한다.

---

## A-7. 채택 / 폐기 / 승격 결정

### 채택
다음 조건을 만족하면 채택 가능:

- 완료 기준 충족
- 주요 테스트 통과
- 플레이 체감 문제 없음
- 성능 허용 범위 내
- 유지 기간이 짧거나 격리 범위 내에서 관리 가능

### 폐기
다음 조건이면 폐기 권장:

- 반복 수정에도 요구사항 안정 충족 실패
- 디버깅에 내부 코드 이해가 필요함
- 예외 케이스가 계속 증가함
- 다른 시스템과 충돌이 심함

### 승격
다음 조건이면 Case B로 전환:

- 기능을 장기 운영해야 함
- 재사용 요구가 생김
- 확장 요청이 붙음
- 코드 이해 없이 유지가 어려움

---

## A-8. 문서화 및 인수인계

### 남겨야 할 기록

- 최종 요구사항
- 테스트 체크리스트
- 채택 버전
- 알려진 한계
- 재작업 필요 조건
- Case B 승격 조건

### 권장 산출물

- Result Report
- QA 결과표
- 플레이 영상 또는 캡처

---

## A-9. Case A 완료 기준

다음 항목을 모두 만족해야 완료로 본다.

- 요구사항 충족
- 지정 테스트 통과
- 빌드/실행 성공
- 성능 기준 충족
- 알려진 리스크 기록 완료
- 채택/폐기/승격 결정 완료

# 8. Case B 파이프라인 - 화이트박스형 코드 리뷰 프로세스

## 개요 흐름

**문제 정의 → 코드 컨텍스트 수집 → 설계안 검토 → AI 구현 → 인간 코드 리뷰 → 테스트/프로파일링 → 문서화 및 머지**

---

## B-1. 변경 대상 정의

### 목표
무엇을 왜 바꾸는지와, 변경이 미치는 범위를 명확하게 한다.

### 수행 내용

- 문제 정의 또는 기능 정의
- 관련 모듈 식별
- 영향 받는 시스템 정리
- 리스크 등급 부여

### 산출물

- 변경 범위 문서
- 영향도 분석
- 리스크 분류

---

## B-2. 코드 컨텍스트 수집

### 목표
AI와 리뷰어가 같은 맥락을 공유하도록 한다.

### 수집 대상

- 관련 파일
- 기존 아키텍처 설명
- 호출 흐름
- 데이터 구조
- 엔진 생명주기 훅
- 테스트 코드
- 기존 버그 재현 절차

### 운영 원칙

- 필요한 파일만 제공한다.
- 무관한 대규모 컨텍스트를 주지 않는다.
- 변경 범위를 명확히 제한한다.

---

## B-3. 구현 전 설계안 작성 및 검토

### 목표
바로 코드를 만들지 말고, 먼저 구조와 접근법을 검토한다.

### 설계안에 포함할 항목

- 문제 원인 가설
- 수정 전략
- 대안 비교
- 영향 받는 시스템
- 테스트 전략
- 롤백 전략

### 승인 기준
다음 질문에 답할 수 있어야 한다.

- 왜 이 구조가 적절한가?
- 기존 규칙과 충돌하지 않는가?
- 이 변경이 추후 기능 확장에 어떤 영향을 주는가?
- 성능/메모리/네트워크 비용은 어떤가?

설계 검토를 생략하면 AI가 “당장 동작하는 코드”는 만들 수 있어도, **팀 코드베이스에 맞는 코드**를 만들 확률이 급격히 낮아진다.

---

## B-4. AI 구현 - 작은 변경 단위로 수행

### 목표
리뷰 가능한 크기의 변경으로 쪼개서 구현한다.

### 권장 단위

- 1개의 명확한 문제 해결
- 1개의 서브시스템 수정
- 1개의 PR로 리뷰 가능한 크기

### 원칙

- 한 번에 너무 많은 파일을 바꾸지 않는다.
- 구조 변경과 기능 변경을 분리한다.
- 리팩터링과 버그 수정을 가능하면 분리한다.
- AI에게 변경 이유를 주석/문서/커밋 메시지 수준으로 설명하게 한다.

---

## B-5. 인간 코드 리뷰

### 목표
AI 산출물을 팀 기준으로 검증하고 리스크를 제거한다.

### 필수 리뷰 항목

#### 구조

- 책임 분리가 적절한가?
- 기존 아키텍처 원칙과 맞는가?
- 중복 구현이 생기지 않았는가?

#### 가독성

- 변수/함수 이름이 명확한가?
- 흐름이 추적 가능한가?
- 숨겨진 사이드이펙트가 없는가?

#### 안정성

- null/예외 처리 누락이 없는가?
- 상태 전이 오류가 없는가?
- 경계 조건 처리가 되어 있는가?

#### 게임 특화 검토

- 프레임 의존 버그가 없는가?
- Tick/Update/Coroutine/Async 남용이 없는가?
- 메모리 할당이 과도하지 않은가?
- 애니메이션, 입력, 물리 타이밍이 어긋나지 않는가?
- 멀티플레이 환경이라면 권한/동기화 문제가 없는가?
- 저장 데이터 호환성을 깨지 않는가?

#### 협업성

- 다른 개발자가 이어서 수정 가능한가?
- 테스트 코드 또는 검증 수단이 충분한가?
- 문서화할 포인트가 명확한가?

### 리뷰 결과 분류

- 승인
- 수정 후 승인
- 재설계 필요
- 반려

---

## B-6. 테스트, 프로파일링, 회귀 검증

### 목표
코드가 “읽기 좋아 보이는 것”이 아니라 실제로 안전한지 검증한다.

### 테스트 항목

- 단위 테스트
- 통합 테스트
- 실제 플레이 테스트
- 회귀 테스트
- 성능 측정
- 로그/에러 확인

### 게임 개발 체크리스트 예시

- 전투 중 입력 누락 없음
- 씬 전환 후 상태 오염 없음
- 오브젝트 풀링/할당 정책 위반 없음
- 프레임 타임 급증 없음
- 플랫폼별 UI/입력 차이 문제 없음

---

## B-7. 문서화 및 머지

### 목표
AI가 만든 코드가 팀의 자산으로 남을 수 있게 한다.

### 필수 문서화

- 변경 목적
- 주요 설계 포인트
- 주의할 제약 사항
- 테스트 결과
- 후속 개선 과제

### 머지 조건

- 코드 리뷰 승인
- 테스트 통과
- 주요 리스크 해소 또는 명시
- 롤백 가능 상태 확보

# 9. A/B 공통 운영 정책

## 9.1 브랜치 정책

### Case A

- 실험 브랜치 또는 샌드박스 브랜치 권장
- 메인 코드베이스 직접 반영 금지
- 채택 전까지 폐기 가능 상태 유지

### Case B

- 기능 브랜치 사용
- PR 기반 리뷰 필수
- 머지 전 회귀 검증 필수

---

## 9.2 작업 크기 정책

AI가 한 번에 처리하는 단위가 커질수록 실패 비용이 커진다.

권장:

- 작은 버그 수정: 즉시 처리 가능
- 중간 기능: 단계 분할 후 처리
- 큰 시스템: 설계 문서 → 하위 태스크 분해 → 순차 처리

---

## 9.3 A에서 B로 승격하는 규칙

다음 중 하나라도 발생하면 Case B로 전환한다.

- 결과물 유지보수 요청이 생김
- 다른 기능이 해당 코드에 의존하게 됨
- 원인 분석을 위해 코드 이해가 필요해짐
- 성능/메모리 문제가 발견됨
- QA 이슈가 반복적으로 발생함
- 동일 기능의 확장 버전이 필요해짐

---

## 9.4 승인 권한

- **기획 승인:** 기획/디자인 책임자
- **기술 승인:** 개발 리드 또는 리뷰 담당자
- **출시 승인:** 프로젝트 책임자

AI는 어떤 단계에서도 승인자가 될 수 없다.

---

## 9.5 금지 사항

- 완료 기준 없이 AI에게 “알아서 구현” 지시
- 핵심 시스템을 Case A로 바로 본선 반영
- 코드 이해 없이 장기 운영 자산으로 채택
- 테스트 없이 “동작하는 것 같음”으로 승인
- 하나의 요청에 설계 변경, 신규 기능, 리팩터링, 버그 수정 전부 혼합

# 12. 추천 운영 모델

실무에서는 다음처럼 운영하는 것을 권장한다.

## 12.1 기본 정책

- **프로토타입 단계:** Case A 중심
- **버티컬 슬라이스 이후:** A와 B 혼합
- **본 생산/라이브 단계:** Case B 중심

## 12.2 권장 기본값

명확히 판단이 안 되면 **Case B를 기본값**으로 한다.  
이유는 게임 개발에서 나중에 발생하는 비용의 대부분이 구현 속도보다 **유지보수와 디버깅 비용**이기 때문이다.

## 12.3 운영 철학

Case A는 “빠르게 만들어 보고 버릴 수 있는 자유”를 얻는 대신,  
코드 자산화 가능성을 일부 포기하는 방식이다.

Case B는 “속도는 조금 느릴 수 있지만 팀 자산으로 남길 수 있는 구현”을 만드는 방식이다.

둘 중 어느 것이 더 좋은 방식이 아니라, **작업 성격에 맞게 선택해야 하는 서로 다른 생산 방식**으로 본다.

# 14. 최종 결론

이 가이드의 핵심은 단순하다.

- **결과만 보면 되는 작업은 Case A**
- **코드까지 책임져야 하는 작업은 Case B**

게임 제작에서 AI는 매우 강력한 가속기지만, 모든 작업을 같은 방식으로 맡기면 품질과 유지보수성이 무너진다.  
따라서 AI 활용의 핵심은 “잘 쓰는 것”이 아니라 **작업의 성격에 따라 통제 방식을 다르게 설계하는 것**이다.

이 문서를 팀 표준으로 삼을 경우, 이후에는 다음 문서를 추가로 만드는 것을 권장한다.

1. 팀 전용 Task Card 템플릿
2. 엔진별 리뷰 체크리스트 (Unity / Unreal / Godot 등)
3. AI 작업 승인 규칙
4. 프로토타입 → 제품 코드 승격 기준
