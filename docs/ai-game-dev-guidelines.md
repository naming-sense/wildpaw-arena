# AI를 활용한 서버/클라이언트 개발 가이드라인

## 1. 문서 목적

이 문서는 **AI를 활용한 서버/클라이언트 개발 작업을 실무적으로 운영하기 위한 기준**을 정의한다.

이번 버전에서는 범위를 넓게 잡지 않고, **게임 서버/클라이언트 개발**에만 집중한다.
또한 A와 B의 구분도 단순히 코드 확인 여부가 아니라, **개발 단계와 협업 맥락**을 기준으로 재정의한다.

- **Case A = 프로토타이핑 / MVP 단계**
- **Case B = 협업 단계 / 메인 프로젝트 마이그레이션 단계**

즉, 이 문서의 핵심은 다음이다.

1. 서버/클라이언트 개발에서 AI를 어떤 단계에 어떻게 써야 하는지 정리한다.
2. 빠르게 검증해야 하는 단계(A)와, 팀 자산으로 편입해야 하는 단계(B)를 구분한다.
3. 프로토타입 코드를 어디까지 허용하고, 언제부터는 구조화/리뷰/테스트를 강화해야 하는지 기준을 만든다.

---

## 2. 적용 범위

이 문서는 다음 작업에 적용한다.

### 클라이언트 개발 범위
- 입력 처리
- 플레이어 이동/전투 동작
- HUD/UI 흐름
- 씬 전환 및 게임플로우
- 클라이언트 상태 관리
- 네트워크 클라이언트 연동
- 임시 디버그 UI / 개발용 툴

### 서버 개발 범위
- 룸/세션 흐름
- 매치메이킹 프로토타입
- 게임 상태 시뮬레이션
- 네트워크 이벤트 처리
- 서버 API / 어드민 엔드포인트
- 로그 / 메트릭 / 디버그 도구
- 서버 설정/런타임 제어 로직

### 제외 또는 별도 엄격 관리가 필요한 영역
다음은 원칙적으로 초기부터 **Case B 수준 이상**으로 다룬다.

- 인증 / 권한
- 결제 / 과금
- 저장 데이터 호환성
- 보안 민감 로직
- 라이브 운영 핵심 장애 대응 로직
- 프로덕션 배포 인프라 핵심 자동화
- 멀티플레이 권한 판정 핵심 로직

---

## 3. A / B 구분 기준

## 3.1 핵심 정의

### Case A: 프로토타이핑 / MVP용 개발
AI를 사용해 **빠르게 동작을 검증하고 방향성을 확인하는 단계**다.

이 단계에서는 다음이 중요하다.

- 빨리 만들어 볼 것
- 실제로 동작하는지 확인할 것
- 기획/플레이/흐름 검증이 가능할 것
- 나중에 버리거나 다시 짤 가능성을 감수할 것

즉, A는 **속도와 검증**이 우선이고, 코드 자산화는 부차적이다.

### Case B: 협업 / 메인 프로젝트 마이그레이션용 개발
AI를 사용하더라도 결과물을 **팀이 읽고, 리뷰하고, 유지보수할 수 있는 코드 자산**으로 만드는 단계다.

이 단계에서는 다음이 중요하다.

- 다른 개발자가 이해할 수 있을 것
- 메인 프로젝트 구조에 맞을 것
- 코드 리뷰가 가능할 것
- 테스트, 디버깅, 운영이 가능할 것
- 향후 기능 확장과 유지보수가 가능할 것

즉, B는 **통합 가능성, 협업 가능성, 유지보수성**이 우선이다.

---

## 3.2 한 줄 기준

### A를 선택하는 경우
- 지금 목표가 **아이디어 검증**이다.
- 빠르게 플레이 가능한 상태를 만드는 것이 우선이다.
- 코드가 나중에 폐기되거나 대체될 수 있다.
- 혼자 실험하거나 소수 인원이 빠르게 검토하면 된다.
- 메인 프로젝트에 바로 합칠 계획이 없다.

### B를 선택하는 경우
- 다른 개발자와 함께 이어서 개발해야 한다.
- 메인 프로젝트 코드베이스에 넣어야 한다.
- 유지보수 기간이 길다.
- 문제가 생겼을 때 다른 사람이 원인 분석을 해야 한다.
- 기능이 확장될 예정이거나 공용 모듈이 된다.

---

## 3.3 기본 원칙

- **프로토타이핑 / MVP는 A를 기본값으로 한다.**
- **협업 / 메인 프로젝트 편입은 B를 기본값으로 한다.**
- 애매하면 “이 코드를 다른 개발자가 이어받아도 되는가?”를 기준으로 판단한다.
- 그 답이 “아직 아니다”면 A, “그래야 한다”면 B다.

---

## 4. Case A - 프로토타이핑 / MVP 개발 프로세스

## 4.1 목적

Case A의 목적은 **빠른 가설 검증**이다.

여기서의 성공은 다음과 같다.

- 서버/클라이언트 기능이 의도한 흐름으로 돌아간다.
- 핵심 플레이 감각이나 기술 가능성을 확인했다.
- MVP 범위에서 데모 가능한 상태가 되었다.
- 다음 단계로 갈 가치가 있는지 판단할 수 있다.

여기서의 성공은 **좋은 구조의 최종 코드**가 아니라, **의사결정을 가능하게 만드는 작동 결과물**이다.

---

## 4.2 Case A에 적합한 작업 예시

### 클라이언트
- 이동/대시/스킬 감각 빠르게 붙여보기
- 임시 HUD와 체력바 표시
- 로비 → 매칭 → 인게임 → 결과 화면 흐름 목업
- 네트워크 수신값으로 UI 반응시키는 데모
- 임시 입력 체계로 플레이 감각 확인

### 서버
- 룸 생성/입장/퇴장 흐름 프로토타입
- 간단한 매칭 큐 실험
- 상태 동기화 방식 샘플 구현
- 더미 AI/Bot 포함 전투 세션 시뮬레이션
- 운영용이 아닌 디버그 API 샘플

---

## 4.3 Case A 작업 원칙

### 원칙 1. 결과 검증이 우선이다
코드가 예쁘게 보이는 것보다, **실제로 원하는 흐름이 동작하는지**가 중요하다.

### 원칙 2. 폐기 가능성을 전제로 한다
A에서 만든 결과물은 **나중에 버릴 수 있다**는 전제를 깔고 간다.

### 원칙 3. 메인 프로젝트 오염을 최소화한다
가능하면 다음 중 하나로 격리한다.

- 별도 브랜치
- 실험용 디렉토리
- 프로토타입 저장소
- feature flag 뒤의 임시 구현

### 원칙 4. 한 번에 큰 시스템 전체를 만들지 않는다
A에서도 작업 단위는 작아야 한다.

- 한 화면 흐름
- 한 입력 루프
- 한 서버 이벤트 흐름
- 한 네트워크 시퀀스

### 원칙 5. 완료 기준은 반드시 적는다
A라고 해서 “대충 되면 됨”으로 가면 안 된다.

---

## 4.4 Case A 작업 파이프라인

### A-1. 검증 목표 정의
먼저 무엇을 검증하려는지 정의한다.

예:
- 이 대시 감각이 재미있는가?
- 서버 authoritative 이동 모델이 지금 구조에서 가능한가?
- 로비-매칭-게임 진입 UX가 끊기지 않는가?
- 5분 플레이 기준으로 MVP 세션이 성립하는가?

### 산출물
- 검증 질문 1~3개
- 데모 성공 기준
- 관찰 포인트

---

### A-2. MVP 범위 고정
기능을 늘리지 말고, 이번 사이클에서 필요한 최소 범위를 고정한다.

예:
- 캐릭터 1종
- 맵 1개
- 룸 타입 1개
- 결과 화면 최소 정보만 표시
- 인증/친구/상점 기능 제외

### 규칙
- “나중에 필요할 것 같아서” 넣지 않는다.
- A에서는 확장성보다 완성 가능한 최소 단위를 택한다.

---

### A-3. AI 요청서 작성
AI에게 줄 요청은 다음 기준으로 구성한다.

- 목표 동작
- 입력/출력
- 제외 범위
- 임시 구현 허용 범위
- 데모 기준
- 테스트 방식

#### A용 요청 예시
```md
목표:
- 클라이언트에서 이동/대시/피격 애니메이션 흐름이 연결된 플레이 가능한 데모를 만든다.

범위:
- 플레이어 1종, 맵 1개, 적 더미 1종
- 임시 UI 허용
- 네트워크는 로컬 mock 허용

완료 기준:
- 3분 동안 조작 가능한 데모가 실행된다.
- 대시 쿨다운이 눈에 보인다.
- 피격 시 애니메이션/HP 감소/UI 반응이 연결된다.

제외:
- 저장/로드
- 실제 멀티플레이 동기화
- 최종 아트 품질
```

---

### A-4. AI 구현 및 빠른 반복
AI에게 구현을 맡기고, 사람은 빠르게 플레이/실행해 본다.

이 단계에서 중요한 것은:

- 코드 완성도보다 시연 가능 상태
- 작은 수정 요청의 빠른 반복
- “무엇이 부족한지”를 플레이 기준으로 피드백

### 좋은 피드백 예시
- “대시는 되는데 선딜 때문에 조작감이 무겁다. 선딜을 절반으로 줄여 달라.”
- “서버 이벤트 순서가 뒤집혀서 클라 UI가 먼저 갱신된다. 결과 UI는 확정 응답 뒤에만 표시해 달라.”
- “세션 종료 후 로비 복귀 흐름이 끊긴다. 자동 복귀 처리 추가.”

---

### A-5. 데모 검수
Case A의 검수는 **플레이/실행 검수** 중심이다.

#### 클라이언트 체크
- 실제 조작이 가능한가?
- 원하는 게임 감각이 나오는가?
- 흐름이 끊기지 않는가?
- 임시 UI라도 정보를 전달하는가?
- 치명적인 프레임 저하가 없는가?

#### 서버 체크
- 세션 흐름이 끊기지 않는가?
- 기본 상태 전이가 정상인가?
- 디버그 로그로 흐름 추적이 가능한가?
- 예상한 입력/응답 순서가 유지되는가?
- 크래시/무한루프/치명적 메모리 문제는 없는가?

---

### A-6. 결과 분류
A 단계 결과는 아래 셋 중 하나로 정리한다.

#### 1) 통과
- 프로토타입 목표를 달성했다.
- MVP 데모로 활용 가능하다.
- 다음 단계로 넘어갈 가치가 있다.

#### 2) 보류
- 방향성은 맞지만 아직 검증이 부족하다.
- 추가 실험이 더 필요하다.

#### 3) 폐기
- 재미/기술/구조 측면에서 가치가 낮다.
- 다른 접근이 더 낫다.

A에서는 **폐기 결정도 성과**다.

---

## 4.5 Case A 산출물

A 단계가 끝나면 최소한 아래는 남겨야 한다.

- 무엇을 검증했는가
- 어떤 범위까지 만들었는가
- 무엇이 동작했는가
- 무엇이 임시 구현인가
- 무엇이 아직 문제인가
- B로 넘길 가치가 있는가

### 권장 출력 포맷
- Prototype Summary
- MVP Demo Checklist
- Known Limitations
- Migration Notes

---

## 4.6 Case A 완료 기준

다음을 만족하면 A 단계 완료로 본다.

- 검증 질문에 대한 답이 나왔다.
- 데모 또는 MVP 실행 결과가 있다.
- 범위 밖 기능이 섞이지 않았다.
- 임시 구현 / 알려진 한계를 기록했다.
- B로 넘길지 여부를 결정했다.

---

## 5. Case B - 협업 / 메인 프로젝트 마이그레이션 프로세스

## 5.1 목적

Case B의 목적은 **AI가 만든 결과를 팀이 유지 가능한 형태로 정제해 메인 프로젝트 자산으로 편입**하는 것이다.

여기서의 성공은 다음과 같다.

- 다른 개발자가 읽고 수정할 수 있다.
- 메인 프로젝트 구조와 규칙에 맞는다.
- 리뷰, 테스트, 디버깅, 운영이 가능하다.
- 향후 기능 확장에 버틸 수 있다.

즉, B는 단순히 “돌아가는 코드”가 아니라 **합류 가능한 코드**를 만드는 단계다.

---

## 5.2 Case B가 필요한 상황

다음 중 하나라도 해당하면 B로 본다.

- 다른 개발자에게 작업을 넘겨야 한다.
- 메인 저장소 / 메인 브랜치로 옮겨야 한다.
- 공용 모듈로 재사용될 예정이다.
- QA/운영/디버깅 체계에 편입해야 한다.
- 프로토타입 코드를 제품 코드로 전환하려 한다.
- 서버/클라이언트 간 계약을 고정해야 한다.

---

## 5.3 Case B 작업 원칙

### 원칙 1. 메인 프로젝트 기준을 우선한다
프로토타입에서 편했던 방식이 메인 프로젝트에 맞지 않으면 바꿔야 한다.

### 원칙 2. 계약과 경계를 먼저 정리한다
특히 서버/클라이언트 개발에서는 아래가 중요하다.

- 이벤트 스펙
- 상태 전이 규칙
- 오류 처리 규칙
- 데이터 구조
- authority 경계
- 재시도/타임아웃 정책

### 원칙 3. 다른 사람이 읽을 수 있어야 한다
코드, 구조, 파일 배치, 네이밍, 로그가 모두 협업 기준을 만족해야 한다.

### 원칙 4. 리뷰 가능 크기로 쪼갠다
A에서 한 번에 크게 만들 수 있었더라도, B에서는 리뷰 가능한 단위로 쪼개야 한다.

### 원칙 5. 테스트와 회귀 확인이 필수다
특히 서버/클라이언트는 “지금은 되는 것처럼 보이는 문제”가 이후 크게 터질 수 있다.

---

## 5.4 Case B 작업 파이프라인

### B-1. 마이그레이션 대상 정의
먼저 A 결과물 또는 신규 기능 중, 무엇을 메인 프로젝트에 편입할지 정의한다.

#### 정리 항목
- 편입 대상 기능
- 제외할 임시 구현
- 다시 작성이 필요한 영역
- 유지할 핵심 로직
- 팀 공통 규칙과 충돌하는 부분

### 질문
- 프로토타입에서 살릴 것은 무엇인가?
- 버릴 것은 무엇인가?
- 메인 프로젝트에 맞추려면 무엇을 다시 설계해야 하는가?

---

### B-2. 서버/클라이언트 계약 정리
협업 단계에서는 구현보다 먼저 **계약 정리**가 중요하다.

#### 정리 대상
- 메시지 타입
- 요청/응답 구조
- 이벤트 순서
- 세션 상태 전이
- 오류 코드
- 재접속/타임아웃 처리
- 버전 호환성 규칙

#### 산출물 예시
- Event Contract 문서
- State Transition 표
- API / Packet 명세
- Client Responsibility / Server Responsibility 구분표

이 단계가 약하면 이후 협업 비용이 급격히 늘어난다.

---

### B-3. 구조 재설계 또는 구조 정리
프로토타입에서 나온 코드를 메인 프로젝트에 넣기 전, 필요한 경우 구조를 다시 잡는다.

#### 클라이언트 예시
- 입력 처리와 연출 로직 분리
- HUD 갱신과 게임 상태 갱신 분리
- 네트워크 수신 핸들러와 UI 반응 분리
- 임시 전역 상태를 정식 상태 관리 계층으로 이동

#### 서버 예시
- 세션 관리와 게임 시뮬레이션 분리
- 매치메이킹과 룸 실행 분리
- 디버그 코드와 운영 코드 분리
- 패킷 파싱과 도메인 처리 분리

---

### B-4. AI 구현 또는 재구현
이 단계에서 AI는 여전히 사용할 수 있지만, 요청 방식이 A와 다르다.

B에서는 AI에게 다음을 요구해야 한다.

- 변경 이유를 설명할 것
- 기존 구조와 맞출 것
- 영향 범위를 명시할 것
- 리뷰 가능한 작은 단위로 제출할 것
- 테스트 포인트를 함께 제안할 것

#### B용 요청 예시
```md
목표:
- 프로토타입 단계에서 검증된 룸 상태 전이 로직을 메인 서버 구조에 맞게 편입한다.

요구사항:
- SessionManager, RoomRuntime, EventDispatcher 역할을 분리할 것
- 기존 packet schema와 호환될 것
- 로그 포맷은 현재 서버 규칙을 따를 것
- 리뷰 가능한 단위로 변경할 것

반드시 포함:
- 변경 의도 설명
- 영향 받는 모듈 목록
- 테스트 포인트
- 임시 구현 제거 여부
```

---

### B-5. 인간 코드 리뷰
B에서는 코드 리뷰가 필수다.

#### 공통 리뷰 항목
- 구조가 메인 프로젝트 규칙에 맞는가?
- 역할 분리가 적절한가?
- 임시 코드가 남아 있지 않은가?
- 네이밍과 파일 구성이 일관적인가?
- 디버깅 포인트가 충분한가?

#### 클라이언트 리뷰 항목
- 입력/연출/UI/네트워크가 뒤섞이지 않았는가?
- 프레임 의존 문제가 없는가?
- 씬 전환 및 상태 정리가 안전한가?
- 로딩/에러/재시도 UX가 정의되어 있는가?

#### 서버 리뷰 항목
- authority 경계가 분명한가?
- 상태 전이가 안정적인가?
- 예외 처리와 로그가 충분한가?
- 동시성/세션 종료/재접속 케이스가 고려되었는가?
- 관측 가능성(log/metric/debug hook)이 확보되었는가?

---

### B-6. 테스트 및 회귀 검증
B에서는 단순 실행 확인이 아니라 **운영 가능한 수준의 검증**이 필요하다.

#### 필수 검증 예시
- 단위 테스트 가능 영역 테스트
- 통합 테스트
- 서버/클라이언트 연결 테스트
- 예외 흐름 테스트
- 회귀 테스트
- 성능 확인

#### 클라이언트 예시
- 씬 재진입 시 상태 초기화 정상
- reconnect 또는 session reset 시 UI 꼬임 없음
- 30분 플레이 기준 메모리/프레임 이상 없음

#### 서버 예시
- 세션 생성/종료 반복 안정성
- 잘못된 패킷 처리 안전성
- 타임아웃/재접속 처리
- 로그/메트릭으로 장애 추적 가능

---

### B-7. 메인 프로젝트 편입
검증이 끝나면 메인 프로젝트에 맞는 절차로 편입한다.

#### 필수 항목
- 브랜치 전략 준수
- PR 또는 리뷰 절차 준수
- 관련 문서 업데이트
- 운영상 주의사항 기록
- 롤백 가능 여부 확인

#### 남겨야 할 것
- 왜 이 구조로 편입했는가
- 프로토타입 대비 무엇이 바뀌었는가
- 아직 남은 기술 부채가 무엇인가
- 다음 개발자가 어디서부터 이어받으면 되는가

---

## 5.5 Case B 완료 기준

다음을 만족하면 B 단계 완료로 본다.

- 메인 프로젝트 기준에 맞게 구조가 정리되었다.
- 서버/클라이언트 계약이 문서화되었다.
- 코드 리뷰를 통과했다.
- 테스트 및 회귀 검증을 마쳤다.
- 협업 가능한 상태로 문서화되었다.
- 메인 저장소 또는 메인 브랜치 편입 준비가 끝났다.

---

## 6. A에서 B로 넘어가는 전환 기준

이 문서에서 가장 중요한 부분 중 하나다.

A는 영원히 유지하는 단계가 아니다.  
다음 조건이 되면 반드시 B로 전환해야 한다.

### 전환 조건
- 프로토타입이 실제 제품 방향으로 채택되었다.
- MVP 결과를 기반으로 계속 확장하기로 결정했다.
- 다른 개발자가 이어받아야 한다.
- 메인 프로젝트와 코드/데이터를 연결해야 한다.
- 서버/클라이언트 계약을 고정해야 한다.
- 디버깅/운영/QA 요구가 붙기 시작했다.

### 전환 신호
- “이제 이거 버리기 아깝다”는 말이 나온다.
- “다른 개발자도 이 코드 알아야 한다”는 말이 나온다.
- “이걸 메인 레포에 넣어야 한다”는 말이 나온다.

이 세 가지 중 하나라도 나오면 사실상 B를 시작해야 한다.

## 7. A와 B의 차이 요약

| 항목 | Case A | Case B |
|---|---|---|
| 개발 맥락 | 프로토타입 / MVP | 협업 / 메인 프로젝트 편입 |
| 최우선 목표 | 빠른 검증 | 유지 가능한 자산화 |
| 코드 품질 기준 | 동작 중심 | 구조/리뷰/유지보수 중심 |
| 허용되는 임시 구현 | 비교적 높음 | 최소화 또는 제거 |
| 브랜치/저장소 | 실험 분리 권장 | 메인 프로젝트 규칙 준수 |
| 테스트 수준 | 데모/핵심 흐름 검증 | 회귀/통합/운영 검증 |
| 문서화 수준 | 검증 결과 기록 | 계약/구조/운영 문서화 |
| 협업 적합성 | 낮아도 됨 | 높아야 함 |
| 메인 프로젝트 반영 | 원칙적으로 바로 하지 않음 | 반영을 전제로 진행 |

## 9. 권장 운영 방식

### 9.1 추천 흐름
실무에서는 다음 순서를 권장한다.

1. **A로 빠르게 검증한다.**
2. 가치가 확인된 것만 남긴다.
3. 남길 기능만 B로 재정리한다.
4. B에서 메인 프로젝트 자산으로 편입한다.

이 흐름이 좋은 이유는 단순하다.

- 아이디어 검증은 빠르게
- 팀 자산화는 신중하게

둘을 한 번에 하려고 하면 속도도 잃고 구조도 망가질 가능성이 높다.

### 9.2 실무 해석
A는 “빨리 만들어서 맞는지 보자” 단계이고,  
B는 “이제 팀이 책임질 코드로 만들자” 단계다.

이 둘을 섞지 않는 것이 중요하다.

## 11. 최종 결론

서버/클라이언트 개발에서 AI 활용을 잘하려면, A와 B를 “코드를 봤냐 안 봤냐”로만 나누기보다 **언제 어떤 목적으로 개발하느냐**로 나누는 것이 더 실무적이다.

- **프로토타입 / MVP를 빨리 만들고 검증하는 단계는 A**
- **협업 가능한 코드로 정리해 메인 프로젝트에 넣는 단계는 B**

즉,

- A는 **속도와 검증의 단계**
- B는 **정리와 편입의 단계**다.

이 기준을 팀 규칙으로 삼으면, AI를 더 공격적으로 활용하면서도 메인 프로젝트 품질은 지킬 수 있다.
