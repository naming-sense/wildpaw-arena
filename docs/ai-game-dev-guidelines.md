# AI를 활용한 서버/클라이언트 개발 운영 가이드라인

## 1. 핵심 요약

이 문서는 게임 서버/클라이언트 개발에서 AI를 실무에 어떻게 적용할지 정리한 요약본이다.

핵심은 두 가지다.

- **A:** 빠르게 검증하는 방식
- **B:** 메인 프로젝트에 편입하는 방식

둘은 완전히 다른 방법이 아니다.  
둘 다 비슷한 **기본 작업 흐름**을 따른다.

1. 요구사항 / 개발 계획 정리
2. 작은 단위 구현
3. AI 리뷰 + 인간 리뷰
4. 테스트 준비
5. 실제 테스트
6. 문제 수정 후 반복

차이는 하나다.

- **A는 문제를 문서와 검증 환경 기준으로 고친다.**
- **B는 문제를 코드와 구조 기준으로 고친다.**

---

## 2. A와 B를 나누는 기준

아래 질문으로 구분하면 된다.

### A로 가야 할 때
- 지금 필요한 것은 **빠른 검증**이다.
- 감각, 흐름, 기술 가능성을 먼저 보고 싶다.
- 코드가 나중에 버려지거나 대체돼도 괜찮다.
- 문제가 생기면 요구사항, 시나리오, 하네스를 고쳐서 다시 확인할 생각이다.

### B로 가야 할 때
- 메인 프로젝트에 넣어야 한다.
- 다른 개발자와 협업해야 한다.
- 코드 리뷰, 테스트, 유지보수가 필요하다.
- 문제가 생기면 실제 코드와 구조를 분석해서 고쳐야 한다.

한 줄로 정리하면:

- **A는 검증용**
- **B는 편입용**

---

## 3. 기본 작업 흐름

A와 B 모두 아래 순서를 기본으로 한다.

### 1) 요구사항 / 개발 계획 정리
먼저 목표, 범위, 제외 범위, 완료 기준, 테스트 포인트를 적는다.

### 2) 작은 단위 구현
AI에게 큰 기능 전체를 한 번에 맡기지 않고, 작은 단위로 나눠 구현한다.

### 3) AI 리뷰 + 인간 리뷰
구현 후 바로 다음 단계로 가지 말고, 누락이나 위험 요소를 먼저 점검한다.

### 4) 테스트 준비
검증에 필요한 환경을 준비한다.

- A: 하네스, mock, replay, 시나리오
- B: 테스트 프레임워크, fixture, contract test, smoke test

### 5) 실제 테스트
실제로 돌려보면서 요구사항 충족 여부, 흐름 문제, 예외 상황, 회귀 가능성을 확인한다.

### 6) 문제 수정 후 반복
문제가 나오면 수정하고 다시 검증한다.

---

## 4. A에서 문제를 푸는 방식

A는 **하네스 엔지니어링 기반 검증 방식**이다.

여기서 중요한 것은 완성 코드보다,
**짧은 시간 안에 여러 번 돌려볼 수 있는 검증 환경**이다.

### A에서 문제를 보면 먼저 확인할 것
1. 요구사항이 모호한가?
2. 개발 계획 범위가 너무 넓은가?
3. 검증 시나리오가 부족한가?
4. 하네스가 문제를 제대로 드러내고 있는가?

### A에서 주로 하는 수정
- 요구사항 문구 수정
- 검증 시나리오 추가
- mock / stub 조정
- 하네스 입력값 수정
- 관찰 포인트 추가
- 다시 실행해서 결과 확인

즉 A는 **문서와 검증 환경을 기준으로 수정하는 방식**이다.

### A를 계속 끌면 안 되는 순간
- 원인 분석에 코드 이해가 필수일 때
- 다른 개발자가 이어받아야 할 때
- 메인 프로젝트 편입을 본격적으로 검토할 때

이때는 B로 넘어가는 게 맞다.

---

## 5. B에서 문제를 푸는 방식

B는 **협업과 메인 프로젝트 편입을 위한 방식**이다.

B에서는 공통 흐름에 들어가기 전에 보통 아래 준비가 먼저 필요하다.

1. 기존 프로젝트 기능별 코드베이스 분석
2. 코딩 컨벤션 / 구조 규칙 파악
3. 현재 프로젝트와 대상 모듈 간 차이 정리
4. 마이그레이션 전략 / 요구사항 / 개발 계획 문서화

### B에서 문제를 보면 먼저 확인할 것
1. 어떤 코드 경로에서 문제가 나는가?
2. 구조, 책임 분리, 상태 전이가 맞는가?
3. 컨벤션이나 계약을 어기고 있지 않은가?
4. 테스트가 부족한가?

### B에서 주로 하는 수정
- 실제 코드 패치
- 구조 조정
- 계약 문서 수정
- 테스트 추가 / 수정
- 리뷰 후 재검증

즉 B는 **코드와 구조를 기준으로 수정하는 방식**이다.

---

## 6. AI를 실무에 적용할 때의 원칙

### 1) AI를 한 번에 다 쓰지 않는다
다음처럼 나눠서 쓰는 편이 안정적이다.

- 분석
- 계획 문서 작성
- 구현
- 리뷰
- 테스트
- 재수정

### 2) 항상 작은 단위로 자른다
AI는 큰 범위를 한 번에 맡기면 품질이 흔들리기 쉽다.

### 3) AI를 코드 생성기로만 보지 않는다
AI는 다음 역할에도 유용하다.

- 코드베이스 분석
- 컨벤션 정리
- 문서 초안 작성
- 리뷰 보조
- 테스트 케이스 제안
- 실패 로그 요약

### 4) A와 B를 섞지 않는다
- A는 검증에 집중
- B는 편입과 유지보수에 집중

이 구분이 흐려지면 속도도 품질도 같이 떨어진다.

## 7. 권장 산출물

### A
- 요구사항 / 검증 질문
- 하네스 설명
- 실행 시나리오
- 결과 요약
- B 전환 시 참고 메모

### B
- Codebase Analysis
- Convention Snapshot
- Migration Plan
- Review Notes
- Test Report

---

## 8. 결론

AI를 활용한 서버/클라이언트 개발에서 중요한 것은,
A와 B를 다른 이름으로 부르는 것이 아니라 **어떤 기준으로 문제를 수정하느냐**를 명확히 하는 것이다.

- **A는 문서와 검증 환경 기준으로 수정한다.**
- **B는 코드와 구조 기준으로 수정한다.**

이 기준만 분명해도 AI를 훨씬 실무적으로 쓸 수 있다.
