# AI를 활용한 서버/클라이언트 개발 운영 가이드라인

## 1. 이 문서의 핵심

이 문서는 게임 서버/클라이언트 개발에서 AI를 **어떻게 실무에 배치할지**를 짧고 명확하게 정리한 요약본이다.

핵심 구분은 두 가지다.

- **Case A:** 하네스 엔지니어링 기반 프로토타이핑 / MVP 검증
- **Case B:** 협업과 메인 프로젝트 마이그레이션을 위한 구조화된 개발

한 줄로 요약하면:

- **A는 빨리 검증하는 단계**
- **B는 팀이 유지할 수 있게 편입하는 단계**

AI는 두 단계에서 역할이 다르다.

- A에서 AI는 **실험 가속기**다.
- B에서 AI는 **분석자 + 계획 보조 + 구현 보조 + 리뷰/테스트 보조**다.

---

## 2. A와 B를 가장 쉽게 구분하는 방법

아래 질문에 답하면 된다.

### A를 써야 할 때
- 지금 필요한 것이 **검증**인가?
- 기능보다 **감각, 흐름, 기술 가능성 확인**이 우선인가?
- 메인 프로젝트에 바로 넣지 않아도 되는가?
- 나중에 버리거나 다시 짜도 괜찮은가?

### B를 써야 할 때
- 이 결과물을 **메인 프로젝트에 넣어야 하는가?**
- 다른 개발자가 **이어받아야 하는가?**
- 코드 리뷰, 테스트, 유지보수가 필요한가?
- 서버/클라이언트 계약을 **고정**해야 하는가?

판단이 애매하면 이 질문 하나로 정리하면 된다.

> **“이 코드를 지금 다른 개발자에게 넘겨도 되는가?”**

- 아직 아니면 **A**
- 그래야 하면 **B**

---

## 3. A/B 차이 한눈에 보기

| 항목 | Case A | Case B |
|---|---|---|
| 목적 | 빠른 검증 | 메인 프로젝트 편입 |
| AI의 주 역할 | 하네스, mock, 프로토타입 생성 | 분석, 문서화, 구현 보조, 리뷰/테스트 보조 |
| 핵심 산출물 | 실행 가능한 검증 환경, 데모, 결과 기록 | 마이그레이션 계획, 리뷰된 코드, 테스트 결과 |
| 코드 기대치 | 동작 중심 | 협업/유지보수 중심 |
| 테스트 방식 | 빠른 반복 검증 | 구조적 테스트 + 실제 테스트 |
| 종료 기준 | 검증 완료 / 폐기 / B 전환 결정 | 리뷰/테스트 통과 후 협업 가능 상태 |

## 4. Case A - 하네스 엔지니어링 기반 프로토타이핑

### A의 핵심 생각
A에서는 기능만 만드는 것이 아니라, 그 기능을 **빠르게 반복 검증할 수 있는 하네스(harness)** 까지 함께 만든다.

즉, A의 목표는 “완성 코드”가 아니라,
**짧은 시간 안에 여러 번 돌려보고 판단할 수 있는 실험 환경**이다.

### A의 기본 순서
1. **검증 질문 정의**  
   - 무엇을 확인하려는지 먼저 적는다.
2. **하네스 경계 설정**  
   - 무엇은 실제로 돌리고, 무엇은 mock/stub으로 대체할지 정한다.
3. **AI에 기능 + 하네스 함께 요청**  
   - 구현만 시키지 말고 실행 방법, 입력 시나리오, 관찰 포인트까지 같이 요청한다.
4. **빠르게 반복 실행**  
   - 감각, 이벤트 흐름, 병목, 실패 케이스를 확인한다.
5. **결과 기록 후 B 전환 여부 결정**  
   - 무엇이 유효했고 무엇이 임시 구현인지 남긴다.

### A에서 반드시 남겨야 할 것
- 무엇을 검증했는가
- 어떤 하네스를 만들었는가
- 어떤 시나리오를 돌렸는가
- 결과가 어땠는가
- 메인 프로젝트로 옮길 때 주의할 점은 무엇인가

### A에서 하지 말아야 할 것
- 검증용 코드를 바로 메인 프로젝트에 넣기
- 하네스 없이 기능만 대충 구현하기
- 결과 기록 없이 “좋아 보인다”로 끝내기

## 5. Case B - 협업 / 메인 프로젝트 마이그레이션

### B의 핵심 생각
B는 A에서 검증된 결과나 신규 기능을,
**기존 프로젝트 구조와 팀 규칙에 맞는 코드 자산으로 바꾸는 단계**다.

즉 B의 핵심은 “구현”보다,
**분석 → 계획 → 작은 단위 구현 → 리뷰 → 테스트 → 반복 개선**의 운영 순서를 지키는 것이다.

### B의 기본 순서
1. **기존 프로젝트 기능별 코드베이스 분석**
2. **코딩 컨벤션 / 구조 규칙 파악**
3. **A 결과물과 현재 프로젝트 간 Gap 분석**
4. **마이그레이션 전략 / 요구사항 / 개발 계획 문서화**
5. **문서 기반의 작은 단위 구현**
6. **AI 리뷰 + 인간 리뷰**
7. **테스트 프레임워크 적용**
8. **실제 테스트 수행**
9. **문제 발생 시 반복 이터레이션**
10. **협업 가능한 상태로 정리 및 인수인계**

### B에서 반드시 기억할 것
- **문서 먼저, 구현은 나중**
- A 결과물을 그대로 옮기지 말고 먼저 Gap을 본다.
- 큰 변경을 한 번에 하지 말고 리뷰 가능한 단위로 쪼갠다.
- AI에게 항상 **변경 이유 / 영향 범위 / 테스트 포인트**를 같이 요구한다.

### B의 최종 목표
- 다른 개발자가 이해할 수 있어야 한다.
- 메인 프로젝트 규칙에 맞아야 한다.
- 테스트와 리뷰를 통과해야 한다.
- 다음 작업자가 바로 이어받을 수 있어야 한다.

## 6. AI를 실제 업무에 적용할 때의 운영 원칙

### 원칙 1. AI를 한 번에 다 쓰지 말고 역할별로 나눠 쓴다
권장 흐름:
- 분석 턴
- 규칙 추출 턴
- 계획 문서 턴
- 구현 턴
- 리뷰 턴
- 테스트 턴
- 이터레이션 턴

### 원칙 2. 항상 작은 단위로 자른다
AI가 잘하는 것은 속도다.  
AI가 약해지는 지점은 너무 큰 범위를 한 번에 맡겼을 때다.

### 원칙 3. A와 B를 섞지 않는다
- A는 검증용
- B는 편입용

이 둘을 섞으면 속도도 잃고 품질도 무너지기 쉽다.

### 원칙 4. AI를 코드 생성기만으로 보지 않는다
AI는 아래 역할에도 강하다.

- 코드베이스 분석
- 컨벤션 요약
- 문서 초안 작성
- 리뷰 보조
- 테스트 케이스 초안 작성
- 실패 로그 요약
- 반복 이터레이션 우선순위 정리

## 7. 추천 산출물

### A 단계
- Harness Summary
- 검증 질문 목록
- 실행 시나리오
- 결과 요약
- Migration Notes

### B 단계
- Codebase Analysis
- Convention Snapshot
- Migration Plan
- Review Notes
- Test Report

---

## 8. 결론

서버/클라이언트 개발에서 AI를 잘 쓰려면,
프로토타입 단계와 메인 프로젝트 편입 단계를 같은 방식으로 다루면 안 된다.

- **A는 하네스 엔지니어링 기반의 빠른 검증 단계**
- **B는 협업 가능한 코드로 바꾸는 운영 단계**

이 기준만 명확히 잡아도,
AI를 더 공격적으로 활용하면서도 메인 프로젝트 품질은 지킬 수 있다.
