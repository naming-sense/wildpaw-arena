# AI를 활용한 서버/클라이언트 개발 운영 가이드라인

## 1. 이 문서의 핵심

이 문서는 게임 서버/클라이언트 개발에서 AI를 실제 업무에 적용할 때,
**A와 B를 어떻게 나누고 어떤 순서로 운영할지**를 정리한 핵심 요약본이다.

이 문서의 핵심 메시지는 단순하다.

- A와 B는 완전히 다른 프로세스가 아니다.
- **둘 다 공통 실행 루프를 가진다.**
- 차이는 **문제가 생겼을 때 무엇을 기준으로 수정하느냐**에 있다.

즉,

- **Case A:** 하네스 엔지니어링 기반 검증 중심 워크플로우
- **Case B:** 코드베이스 분석과 마이그레이션 중심 워크플로우

둘 다 다음 흐름을 공유한다.

1. 요구사항 / 개발 계획 문서화
2. 문서 기반의 작은 단위 구현
3. AI 리뷰 + 인간 리뷰
4. 테스트 준비
5. 실제 테스트 수행
6. 문제 발생 시 반복 이터레이션

하지만 문제를 해결하는 방식은 다르다.

- **A는 요구사항, 개발계획, 하네스를 기준으로 수정한다.**
- **B는 코드, 구조, 컨벤션, 테스트를 기준으로 수정한다.**

---

## 2. A와 B를 가장 쉽게 구분하는 방법

아래 질문 하나로 거의 정리된다.

> **“문제가 생겼을 때, 나는 코드를 직접 분석해서 고칠 것인가, 아니면 요구사항과 검증 시나리오를 다시 정리해서 고칠 것인가?”**

### A를 선택하는 경우
- 지금 목적은 **빠른 검증**이다.
- 기능의 감각, 흐름, 기술 가능성을 보고 싶다.
- 코드 자체는 아직 폐기되거나 대체될 수 있다.
- 문제 발생 시 주로 **문서/하네스/시나리오를 바꿔서 다시 돌릴 생각**이다.

### B를 선택하는 경우
- 메인 프로젝트에 넣어야 한다.
- 다른 개발자와 협업해야 한다.
- 유지보수와 코드 리뷰가 필요하다.
- 문제 발생 시 **실제 코드와 구조를 분석해서 수정할 생각**이다.

한 줄로 말하면:

- **A는 검증용**
- **B는 편입용**

---

## 3. 공통 실행 루프

A와 B는 아래 실행 루프를 공통으로 가진다.

### 1) 요구사항 / 개발 계획 문서화
먼저 무엇을 만들고 무엇을 검증할지 문서로 정리한다.

여기에는 최소한 다음이 들어가야 한다.
- 목표
- 범위
- 제외 범위
- 완료 기준
- 테스트 포인트
- 현재 사이클에서 다룰 변경 단위

### 2) 문서 기반의 작은 단위 구현
AI에게 한 번에 큰 범위를 맡기지 않고,
리뷰와 테스트가 가능한 작은 단위로 쪼개서 구현한다.

### 3) AI 리뷰 + 인간 리뷰
구현이 끝나면 바로 다음 단계로 가지 말고,
먼저 AI 리뷰와 사람 리뷰를 통해 누락, 구조 문제, 테스트 포인트를 확인한다.

### 4) 테스트 준비
이 단계에서 필요한 테스트 기반을 준비한다.

- A에서는 주로 하네스, mock, replay, 시나리오 실행기
- B에서는 테스트 프레임워크, fixture, contract test, smoke test

### 5) 실제 테스트 수행
실제로 돌려보면서 다음을 확인한다.
- 요구사항 충족 여부
- 흐름/상태 전이 문제
- 예외 상황 처리
- 회귀 가능성
- 성능/안정성 이슈

### 6) 문제 발생 시 반복 이터레이션
문제가 나오면 수정하고 다시 검증한다.

중요한 것은 여기서부터 A와 B가 달라진다는 점이다.

---

## 4. A와 B의 진짜 차이: 문제를 해결하는 방식

| 항목 | Case A | Case B |
|---|---|---|
| 문제를 보는 기준 | 결과, 동작, 검증 시나리오 | 코드, 구조, 계약, 테스트 |
| 수정의 출발점 | 요구사항 문서 / 개발 계획 / 하네스 | 코드베이스 / 컨벤션 / 마이그레이션 문서 |
| AI에게 주는 피드백 | “결과가 어떻게 달라져야 하는가” | “어떤 코드 경로와 구조를 어떻게 바꿔야 하는가” |
| 코드 해석 수준 | 최소화 | 적극적 |
| 반복 방식 | 문서와 시나리오를 고치고 다시 생성/검증 | 코드 수정, 리뷰, 테스트 추가 후 재검증 |
| 전환 신호 | 코드 이해가 필요해지는 순간 B로 승격 | 계속 B 안에서 구조적으로 해결 |

이 표가 이 문서에서 가장 중요한 부분이다.

즉 A와 B는 둘 다 반복 개발을 한다.  
하지만 **A는 문서/하네스 중심 반복**, **B는 코드/테스트 중심 반복**이다.

---

## 5. Case A 워크플로우

## 5.1 A의 목적
A의 목적은 **빠른 검증**이다.

여기서 중요한 것은 완성 코드가 아니라,
**짧은 시간 안에 여러 번 돌려볼 수 있는 검증 환경**이다.

그래서 A에서는 하네스 엔지니어링이 중요하다.

### 하네스란?
특정 기능이나 흐름을 메인 프로젝트 전체와 분리해서,
반복 실행하고 결과를 관찰할 수 있게 만드는 검증용 실행 환경이다.

---

## 5.2 A의 준비 단계
A를 시작할 때 먼저 정리해야 하는 것은 다음이다.

1. 무엇을 검증할 것인가
2. 어떤 하네스로 검증할 것인가
3. 어떤 결과가 나오면 통과인가
4. 어떤 문제는 무시 가능하고 어떤 문제는 치명적인가

즉 A에서는 구현 전부터 **검증 질문과 하네스 경계**가 먼저 있어야 한다.

---

## 5.3 A의 문제 해결 방식
A에서 문제가 생기면, 먼저 코드를 파고드는 것이 아니라 다음 순서로 본다.

1. **요구사항이 모호한가?**
2. **개발 계획이 너무 넓거나 잘못 잘렸는가?**
3. **하네스 시나리오가 부족한가?**
4. **검증 기준이 불명확한가?**
5. 위를 정리했는데도 안 되면 그때 B 전환을 검토한다.

즉 A에서는 보통 이렇게 수정한다.

- 요구사항 문구 수정
- 개발 계획 범위 재조정
- 입력 시나리오 수정
- mock/stub 조정
- 관찰 포인트 추가
- 하네스 재실행

### A에서의 좋은 질문 방식
- “이 코드 왜 이렇게 짰어?”보다
- “이 시나리오에서 원하는 결과가 안 나온다. 요구사항을 반영해 다시 맞춰줘.”

### A에서 B로 넘어가야 하는 순간
다음 중 하나라도 해당하면 A로 계속 끌지 말고 B로 전환하는 것이 낫다.

- 원인 분석에 코드 이해가 필수다.
- 다른 개발자가 이어받아야 한다.
- 메인 프로젝트 편입 가능성을 본격적으로 검토해야 한다.
- 반복 수정이 문서 수준이 아니라 구조 수준으로 넘어갔다.

---

## 6. Case B 워크플로우

## 6.1 B의 목적
B의 목적은 검증된 결과나 필요한 기능을,
**팀이 유지하고 확장할 수 있는 코드 자산**으로 만드는 것이다.

즉 B는 단순 구현이 아니라,
**분석 → 계획 → 구현 → 리뷰 → 테스트 → 반복 개선**을 프로젝트 기준에 맞게 운영하는 단계다.

---

## 6.2 B의 준비 단계
B는 공통 실행 루프에 들어가기 전에,
아래 준비 작업이 거의 필수다.

1. **기존 프로젝트 기능별 코드베이스 분석**
2. **코딩 컨벤션 / 구조 규칙 파악**
3. **현재 프로젝트와 대상 모듈 간 Gap 분석**
4. **마이그레이션 전략 / 요구사항 / 개발 계획 문서화**

즉 B에서는 구현 전에 이미 **이 프로젝트가 어떻게 동작하는지**를 먼저 이해해야 한다.

---

## 6.3 B의 문제 해결 방식
B에서 문제가 생기면, 문서만 다시 쓰고 끝내지 않는다.  
실제 코드와 구조를 본다.

보통 다음 순서로 해결한다.

1. 어떤 코드 경로에서 문제가 나는지 찾는다.
2. 구조, 책임 분리, 계약, 상태 전이를 점검한다.
3. 필요한 경우 마이그레이션 문서를 수정한다.
4. 작은 단위로 코드를 수정한다.
5. AI 리뷰 + 인간 리뷰를 다시 거친다.
6. 테스트를 다시 돌린다.

즉 B에서는 이렇게 수정한다.

- 실제 코드 패치
- 구조 변경
- 컨벤션 정렬
- 테스트 추가/수정
- 계약 문서 업데이트
- 리뷰 후 재검증

### B에서의 좋은 질문 방식
- “결과만 맞게 해줘”보다
- “현재 코드베이스 구조를 유지하면서 이 경로의 상태 전이 문제를 수정하고 테스트를 보강해줘.”

---

## 7. AI를 실제 업무에 적용할 때의 운영 원칙

### 원칙 1. AI 역할을 턴별로 분리한다
권장 흐름:
- 분석 턴
- 계획 문서 턴
- 구현 턴
- 리뷰 턴
- 테스트 턴
- 이터레이션 턴

### 원칙 2. 항상 작은 단위로 자른다
AI는 큰 목표를 한 번에 맡기면 흔들린다.
작은 단위로 나눌수록 실제 업무 적용성이 높아진다.

### 원칙 3. 리뷰와 테스트는 생략하지 않는다
A든 B든,
문서화 → 구현 → 리뷰 → 테스트 → 반복 이터레이션의 루프는 유지해야 한다.

### 원칙 4. AI를 코드 생성기만으로 쓰지 않는다
AI는 아래 업무에도 강하다.
- 코드베이스 분석
- 컨벤션 정리
- 문서 초안 작성
- 리뷰 보조
- 테스트 케이스 제안
- 실패 로그 요약
- 반복 수정 우선순위 정리

## 8. 권장 산출물

### A에서 남길 것
- 요구사항 / 검증 질문
- 하네스 설명
- 실행 시나리오
- 결과 요약
- B 전환 시 참고할 메모

### B에서 남길 것
- Codebase Analysis
- Convention Snapshot
- Migration Plan
- Review Notes
- Test Report

---

## 9. 결론

AI를 활용한 서버/클라이언트 개발은,
A와 B를 완전히 다른 방법론으로 볼 필요는 없다.

둘은 같은 공통 실행 루프를 공유한다.

- 문서화
- 작은 단위 구현
- 리뷰
- 테스트 준비
- 실제 테스트
- 반복 이터레이션

진짜 차이는 단 하나다.

- **A는 문제를 문서와 하네스 기준으로 해결한다.**
- **B는 문제를 코드와 구조 기준으로 해결한다.**

이 기준만 명확히 잡아도,
AI를 더 실무적으로, 더 안정적으로 사용할 수 있다.
