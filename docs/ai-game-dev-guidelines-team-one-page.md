# AI를 활용한 서버/클라이언트 개발 가이드라인
## 팀 공유용 1페이지 요약

## 1. 핵심 개념
이 문서에서 AI 활용 방식은 두 가지로 나눈다.

- **A:** 빠르게 검증하는 방식
- **B:** 메인 프로젝트에 편입하는 방식

가장 큰 차이는 **문제가 생겼을 때 무엇을 기준으로 수정하느냐**다.

- **A:** 문서, 시나리오, 하네스 기준으로 수정
- **B:** 코드, 구조, 테스트 기준으로 수정

한 줄 요약:
- **A는 검증용**
- **B는 편입용**

---

## 2. A와 B를 나누는 기준

### A를 선택하는 경우
- 지금 필요한 것은 빠른 검증이다.
- 감각, 흐름, 기술 가능성을 먼저 보고 싶다.
- 코드가 나중에 버려지거나 대체돼도 괜찮다.
- 문제가 생기면 요구사항, 시나리오, 하네스를 고쳐서 다시 확인할 생각이다.

### B를 선택하는 경우
- 메인 프로젝트에 넣어야 한다.
- 다른 개발자와 협업해야 한다.
- 코드 리뷰, 테스트, 유지보수가 필요하다.
- 문제가 생기면 실제 코드와 구조를 분석해서 고쳐야 한다.

---

## 3. 기본 작업 흐름
A와 B 모두 아래 순서를 기본으로 한다.

1. **요구사항 / 개발 계획 정리**
2. **작은 단위 구현**
3. **AI 리뷰 + 인간 리뷰**
4. **테스트 준비**
5. **실제 테스트**
6. **문제 수정 후 반복**

---

## 4. A에서 일하는 방식
A는 **하네스 엔지니어링 기반 검증 방식**이다.

### A에서 먼저 보는 것
- 요구사항이 명확한가?
- 검증 시나리오가 충분한가?
- 하네스가 문제를 드러내고 있는가?

### A에서 주로 하는 수정
- 요구사항 문구 수정
- 검증 시나리오 추가
- mock / stub 조정
- 하네스 입력값 수정
- 다시 실행해서 결과 확인

즉 A는 **검증 환경을 다듬어 가며 결과를 맞추는 방식**이다.

---

## 5. B에서 일하는 방식
B는 **협업과 메인 프로젝트 편입 방식**이다.

### B를 시작하기 전에 필요한 것
1. 기존 프로젝트 기능별 코드베이스 분석
2. 코딩 컨벤션 / 구조 규칙 파악
3. 현재 프로젝트와 대상 모듈 간 차이 정리
4. 마이그레이션 전략 / 요구사항 / 개발 계획 문서화

### B에서 먼저 보는 것
- 어떤 코드 경로에서 문제가 나는가?
- 구조, 책임 분리, 상태 전이가 맞는가?
- 컨벤션이나 계약을 어기고 있지 않은가?
- 테스트가 부족한가?

### B에서 주로 하는 수정
- 실제 코드 패치
- 구조 조정
- 계약 문서 수정
- 테스트 추가 / 수정
- 리뷰 후 재검증

즉 B는 **코드와 구조를 다듬어 가며 프로젝트에 맞추는 방식**이다.

---

## 6. AI를 실무에 쓰는 운영 원칙
- AI에게 큰 범위를 한 번에 맡기지 않는다.
- 분석 / 계획 / 구현 / 리뷰 / 테스트를 나눠서 쓴다.
- A와 B를 섞지 않는다.
- AI를 코드 생성기만이 아니라 분석, 문서화, 리뷰, 테스트 보조 도구로 쓴다.

---

## 7. 팀에서 기억할 한 문장
**A는 문서와 하네스 중심으로 검증하고, B는 코드와 구조 중심으로 편입한다.**
