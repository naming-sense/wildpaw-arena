# AI 에이전트를 활용한 게임 개발 가이드라인 - 사용 케이스 / 적용 예시

이 문서는 `ai-agent-game-dev-guidelines.md`의 보조 문서다.  
메인 문서가 핵심 요약본이라면, 이 문서는 **스펙 검증 개발과 마이그레이션 개발이 실제로 어떻게 다르게 작동하는지**를 짧게 보여준다.

---

## 1. 먼저 기억할 점

이 문서의 전제는 단순하다.

- AI는 블랙박스 자동 구현 도구가 아니다.
- AI를 쓰는 목적은 결과물만 받는 것이 아니라,
  내가 더 빨리 이해하고 더 많이 배우는 데 있다.
- 다만 마이그레이션 개발에서는 AI가 항상 더 빠른 것은 아니며,
  직접 구현하는 편이 더 효율적인 순간도 있다.

그럼에도 AI를 같이 쓰는 이유는,
분석, 문서화, 리뷰, 테스트, 비교 검토에서 장기적으로 도움이 크기 때문이다.

---

## 2. 기본 작업 흐름

A와 B는 둘 다 같은 기본 작업 흐름을 따른다.

1. 요구사항 / 개발 계획을 md 파일로 정리
2. md 파일을 기준으로 작은 단위 구현
3. AI 리뷰, 인간 리뷰, 테스트 진행
4. 수정사항 정리 후 다시 1번으로

차이는 문제를 고치는 기준이다.

- **A:** 스펙 문서 기준으로 수정
- **B:** 코드와 구조 기준으로 수정

---

## 3. 같은 문제를 A와 B에서 다르게 푸는 예시

## 예시 1. 결과 UI가 너무 빨리 뜨는 문제

### A(스펙 검증 개발)에서는
- 요구사항에 “결과 UI는 서버 확정 후에만 노출”을 추가한다.
- 테스트 시나리오에 이벤트 순서 확인 항목을 넣는다.
- 문서를 수정한 뒤 다시 구현하고 다시 확인한다.

즉, **원하는 동작을 문서에 더 정확히 적고 다시 검증**한다.

### B(마이그레이션 개발)에서는
- 결과 UI를 띄우는 코드 경로를 찾는다.
- 서버 확정 이벤트와 상태 갱신 순서를 점검한다.
- 필요하면 계약 문서와 상태 전이를 수정한다.
- 코드 패치 후 리뷰와 테스트를 진행한다.

즉, **실제 코드와 구조를 수정해서 해결**한다.

---

## 예시 2. 룸 종료 후 로비 복귀가 가끔 실패하는 문제

### A(스펙 검증 개발)에서는
- 종료 시나리오를 문서에 추가한다.
- 요구사항을 더 명확히 적는다.
- 테스트용 데이터나 입력값을 조정해 다시 확인한다.

### B(마이그레이션 개발)에서는
- 세션 종료 처리 코드와 상태 초기화 경로를 본다.
- cleanup 누락, 이벤트 순서, 상태 전이 문제를 수정한다.
- 테스트를 추가하고 다시 검증한다.

---

## 예시 3. 특정 패킷 순서에서 UI가 꼬이는 문제

### A(스펙 검증 개발)에서는
- 문제 순서를 테스트 시나리오로 정리한다.
- 문서에 예외 케이스를 추가한다.
- 그 기준으로 다시 구현하고 다시 확인한다.

### B(마이그레이션 개발)에서는
- 패킷 처리 핸들러와 상태 갱신 순서를 분석한다.
- 계약 또는 코드 구조를 수정한다.
- 테스트에 편입해서 다시 검증한다.

---

## 4. A에서 AI를 쓰는 방법

A에서는 AI에게 최종 제품 코드를 요구하기보다,
**문서를 빠르게 구현하고 검증하는 데** 집중하는 것이 좋다.

주로 맡길 것:
- 요구사항 문서 초안 작성
- 테스트 시나리오 정리
- 작은 단위 구현
- 결과 비교 정리
- 수정사항 반영

즉 A에서 AI는 **스펙 검증 속도를 올리는 도구**다.

---

## 5. B에서 AI를 쓰는 방법

B에서는 AI를 단순 생성기가 아니라,
**분석 + 문서화 + 구현 + 리뷰 + 테스트 보조 도구**로 쓰는 것이 좋다.

주로 맡길 것:
- 기능별 코드베이스 분석
- 코딩 컨벤션 / 구조 규칙 정리
- 현재 프로젝트와 대상 모듈 간 차이 정리
- 마이그레이션 전략 문서 초안
- 작은 단위 구현 보조
- 리뷰 포인트 정리
- 테스트 포인트 제안
- 실패 로그 요약

즉 B에서 AI는 **마이그레이션 개발 업무를 정리하고 가속하는 도구**에 가깝다.

---

## 6. 마지막 정리

이 문서의 핵심은 간단하다.

- AI는 내 지식베이스를 넓히는 도구로 써야 한다.
- A와 B는 기본 작업 흐름은 같다.
- 차이는 **무엇을 기준으로 수정하느냐**다.

즉,

- **A는 스펙 문서를 수정하면서 검증**
- **B는 기존 프로젝트 분석과 문서화를 바탕으로 코드와 구조를 수정하며 마이그레이션**

이렇게 보면 AI를 훨씬 현실적인 방식으로 운영할 수 있다.
