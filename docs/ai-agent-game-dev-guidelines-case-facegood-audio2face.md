# AI 에이전트를 활용한 게임 개발 가이드라인 - 실제 적용 사례
## FACEGOOD Audio2Face 기반 온디바이스 립싱크 모델 개발과 마이그레이션

## 1. 이 사례가 좋은 예시인 이유

이 사례는 `AI 에이전트를 활용한 게임 개발 가이드라인`에서 말하는 두 가지 방식을 모두 잘 보여준다.

- **A: 스펙 검증 개발**
- **B: 마이그레이션 개발**

처음에는 외부 오픈소스 레포를 가져와서,
**이 기술이 실제로 언리얼 온디바이스 환경에서 쓸 수 있을지 검증**하는 데서 시작했다.
그 과정에서는 요구사항과 계획을 정리하고, AI 에이전트와 대화하면서 구현을 진행했다.

이후 어느 정도 품질이 나오는 결과물을 만든 뒤에는,
기존 프로젝트에 맞게 구조를 맞추고, 코드와 인터페이스를 리뷰하면서
**실제 프로젝트에 편입하는 단계**로 넘어갔다.

즉 이 사례는,

- A 단계에서 어떻게 스펙 검증 개발이 작동하는지
- B 단계에서 어떻게 마이그레이션 개발로 넘어가는지
- 그리고 그 과정에서 AI를 어떻게 학습 도구이자 개발 파트너로 쓸 수 있는지

를 실제로 보여주는 사례다.

---

## 2. 사례 개요

출발점은 다음과 같았다.

- 대상 레포: `FACEGOOD/FACEGOOD-Audio2Face`
- 목적: 예전 기술이지만 **언리얼에서 온디바이스 립싱크로 활용 가능할지 검증**
- 초기 가설: 기존 레포를 그대로 쓰는 것이 아니라,
  - **중국어 데이터셋 → 한국어 데이터셋**
  - **FACEGOOD 112 출력 포맷 → ARKit 52 출력 포맷**
  으로 바꾸면 실제 프로젝트에 더 가까운 형태로 사용할 수 있을 것이라고 판단

즉 처음부터 “레포를 그대로 실행한다”가 아니라,
**실제 사용 가능성을 보기 위해 목표 스펙을 다시 정의한 것**이 핵심이었다.

---

## 3. 이 사례에서 AI를 대하는 방식

이 사례에서 중요한 점은,
AI를 단순 자동 구현 도구로만 쓰지 않았다는 것이다.

물론 실제로는 코덱스와 요구사항을 계속 주고받으며,
코드를 거의 직접 보지 않고 구현이 진행된 구간이 있었다.
하지만 이 과정의 핵심 성과는 단순히 “코드가 나왔다”가 아니다.

더 중요한 성과는 다음이었다.

- 레포의 기술 스택을 빠르게 이해했다.
- 데이터 포맷과 출력 포맷의 차이를 파악했다.
- 모델 구조 변경의 가능성과 한계를 이해했다.
- LPC가 무엇을 하는지, 어느 단계에 들어가는지 이해했다.
- 온디바이스 추론에서 무거운 모델이 갖는 한계를 이해했다.
- 실제 프로젝트 편입 전에 무엇을 먼저 해야 하는지 감이 생겼다.

즉,
이 사례는 **AI를 통해 내 지식베이스를 넓힌 사례**라는 점에서 특히 의미가 크다.

---

## 4. 기술 이해 정리 - LPC에 대한 보정

사용 중에 스스로 정리한 이해 중 핵심은 이 부분이다.

> 이 레포에서는 처음 음성 데이터를 LPC라는 기존 음성 분석 기술로 처리하고,
> 이것은 신경망이 아니라 기존 로직으로 돌아가는 형태다.

이 이해는 **대체로 맞다**.

공개 README 기준으로 보면,
이 레포는 입력 음성에 대해 먼저 **fixed-function analysis**를 수행하고,
`step1_LPC.py`를 통해 wav를 전처리해서 학습 입력으로 사용한다.

즉 이 프로젝트에서 LPC는 대략 다음처럼 이해하면 된다.

- 신경망 자체가 아니라 **전통적인 음성 분석 / 신호처리 단계**에 가깝다.
- 음성을 프레임 단위 특징으로 바꾸는 **전처리 역할**을 한다.
- 따라서 “기존 로직으로 돌아가는 형태”라는 이해는 맞는 편이다.

이걸 이해한 뒤,
“이 전처리 단계를 wav2vec / HuBERT 같은 더 현대적인 표현 추출기로 바꿀 수 있지 않을까?”라는 질문으로 이어진 것도 매우 자연스러운 흐름이었다.

---

## 5. A 단계 - 스펙 검증 개발

이 사례의 초반부는 전형적인 **스펙 검증 개발**이다.

### 5.1 시작 스펙
처음에 정한 핵심 스펙은 대략 다음과 같다.

- 이 레포를 분석해서 실제 적용 가능성을 판단한다.
- 중국어 데이터셋 의존성을 줄이고 한국어 데이터셋으로 바꾼다.
- 기존 출력 포맷 대신 ARKit 52 포맷으로 바꾼다.
- 언리얼과 통신해서 캐릭터 입모양을 움직일 수 있는 수준까지 만든다.
- 온디바이스 사용 가능성도 계속 염두에 둔다.

즉 이 단계의 목적은 “완성 제품 개발”이 아니라,
**기술 가능성과 품질 가능성을 검증하는 것**이었다.

---

### 5.2 AI가 먼저 해준 일
A 단계에서 AI는 먼저 레포 분석 역할을 수행했다.

예를 들어 다음을 빠르게 파악하도록 했다.

- 기술 스택은 무엇인가
- 어떤 입력 데이터를 쓰는가
- 학습/추론 파이프라인은 어떻게 돌아가는가
- 출력 포맷은 무엇인가
- 현재 구조에서 어떤 부분이 병목이 될 수 있는가
- 어떤 부분은 바꾸기 쉽고 어떤 부분은 어렵나

이 단계에서 얻은 가장 큰 이점은,
코드를 하나하나 수동으로 파고들기 전에 **문제의 지형도**를 먼저 잡았다는 점이다.

---

### 5.3 스펙 확장 - 모델 고도화 가능성 탐색
그다음에는 단순 포맷 변환을 넘어서,
모델 자체를 더 좋아지게 만들 수 있는지도 탐색했다.

여기서 AI가 제안한 아이디어는 대략 다음과 같다.

- LPC 기반 입력 대신
- wav2vec, HuBERT 같은 트랜스포머 기반 표현 추출기를 쓰면
- 입모양 품질이 더 좋아질 수 있다

이 제안은 단순 구현 지시가 아니라,
**현재 모델의 한계와 현대적 대안을 비교하는 학습 포인트**가 되었다.

하지만 동시에 AI는 중요한 현실적인 조언도 했다.

- LPC를 버리고 더 큰 모델로 가면
- 모바일 온디바이스 추론에서는 너무 무거워질 수 있다
- 따라서 품질 향상과 추론 비용 사이의 trade-off를 봐야 한다

이 지점이 이 사례의 핵심이다.

A 단계에서 중요한 것은 당장 모든 걸 다 구현하는 것이 아니라,
**무엇이 더 좋은 방향처럼 보이지만 지금 우선순위는 아닌지 판단하는 것**이다.

그래서 모델 고도화는 뒤로 미루고,
우선순위를 다음으로 잡았다.

1. ARKit 52 포맷으로 변환
2. 한국어 데이터셋으로 재학습
3. 언리얼과 연결해서 실제 동작 확인

이 판단 자체가 스펙 검증 개발의 좋은 예다.

---

### 5.4 실제 구현과 검증
이후 실제로 진행한 작업은 다음과 같다.

- NVIDIA NIM 등을 이용해 음성을 **ARKit 52 데이터가 담긴 CSV** 형태로 변환
- 이를 기반으로 **한국어 데이터셋 구성**
- 재학습 진행
- 추론 서버 구현
- 언리얼과 통신해 캐릭터 입모양 구동 성공

이 시점까지는,
요구사항과 계획을 계속 정리하고 코덱스와 대화하면서 진행했으며,
**코드를 한 줄도 직접 보지 않고 구현이 진행된 구간이 많았다**고 볼 수 있다.

그런데도 이 단계가 의미 있었던 이유는,
단순 자동 생성이 아니라 **검증 목표가 명확했기 때문**이다.

즉 “어떤 결과가 나와야 성공인가”가 분명했다.

- 한국어 데이터셋으로 학습 가능할 것
- ARKit 52로 출력할 것
- 언리얼에서 실제로 입모양이 움직일 것
- 온디바이스 적용 가능성을 계속 검토할 것

---

### 5.5 A 단계에서 넓어진 지식베이스
이 사례에서 A 단계의 가장 큰 성과는 결과물만이 아니라,
다음 지식이 실제로 내 머릿속에 들어왔다는 점이다.

- LPC가 신경망이 아니라 전처리/분석 단계라는 점
- LPC 처리에서 고정된 지연 특성이 있다는 점
- 레거시 DLL 형태로 제공되는 처리 로직의 한계
- 이를 대체하기 위해 동적으로 처리 가능한 LPC를 네이티브로 재구현할 수 있다는 점
- 학습 과정에서 누락되는 값들이 입모양 디테일에 직접 영향을 줄 수 있다는 점
- 모델 품질 향상은 아키텍처, 데이터, 파라미터, 후처리 모두와 연결되어 있다는 점
- TensorFlow 기반 코드를 PyTorch로 옮기는 작업 자체도 실무적으로 큰 과제라는 점

즉 A 단계는 단순히 결과를 만든 단계가 아니라,
**기술을 이해하고 문제를 분해할 수 있는 수준까지 올라간 단계**였다.

---

## 6. A에서 B로 넘어간 이유

A 단계가 성공적으로 끝났다고 볼 수 있었던 기준은 분명했다.

- 언리얼에서 실제로 동작했다.
- 어느 정도 품질이 나왔다.
- 기술 가능성이 확인되었다.
- 이제는 “되는가?”보다 “기존 프로젝트에 어떻게 넣을 것인가?”가 더 중요한 문제가 되었다.

이 시점이 바로,
**스펙 검증 개발에서 마이그레이션 개발로 넘어가는 전환점**이다.

---

## 7. B 단계 - 마이그레이션 개발

B 단계에서는 더 이상 “될까?”만 보는 것이 아니라,
**기존 프로젝트 구조에 어떻게 안전하게 넣을 것인가**가 핵심이 된다.

### 7.1 마이그레이션 계획 수립
이 사례에서는 기존 프로젝트에 바로 붙인 것이 아니라,
먼저 편입 계획을 세웠다.

핵심 방향은 다음과 같았다.

- `AICenter`가 제공하는 플러그인에 Audio2Face를 통합한다.
- 기존에 동작하던 레거시 립싱크는 **fallback**으로 유지한다.
- 그 위에 Audio2Face 기반 기능이 동작하도록 설계한다.

이건 매우 좋은 마이그레이션 전략이다.

왜냐하면 새 기능을 넣는 동시에,
기존 기능을 완전히 버리지 않고 안전장치를 남겨두었기 때문이다.

---

### 7.2 B 단계에서의 작업 방식 변화
A 단계와 가장 큰 차이는 여기서 발생한다.

A에서는 요구사항과 결과 중심으로 갔지만,
B에서는 **코드를 리뷰하고 이해하면서 진행**했다.

즉 B 단계에서는 다음을 실제로 보기 시작했다.

- 기존 코딩 컨벤션과 어긋나는 점은 없는가
- 플러그인 인터페이스와 잘 맞는가
- 기존 시스템과 책임 분리가 맞는가
- fallback 구조가 안정적으로 동작하는가
- 이후 유지보수 가능한 형태인가

이 지점이 바로 마이그레이션 개발의 본질이다.

즉 B는 AI가 대신 만들어주는 단계를 넘어서,
**내가 코드와 구조를 이해하고 프로젝트 기준으로 판단하는 단계**다.

---

### 7.3 편입 이후 확장 작업
B 단계에서는 단순 통합만 한 것이 아니라,
실제 프로젝트 가치를 높이는 기능 확장도 진행했다.

예를 들어:

- 에디터에서 베이킹할 수 있게 함
- 베이킹된 데이터를 사용해 추론 없이 바로 시퀀서에서 사용할 수 있게 함

이건 단순 기능 이식이 아니라,
기존 프로젝트 워크플로우에 맞춘 **실전형 확장**이다.

즉 마이그레이션 개발이 잘 되었기 때문에,
이후 기능이 프로젝트 문맥 안에서 자연스럽게 확장될 수 있었던 것이다.

---

## 8. 이 사례가 가이드라인과 정확히 맞아떨어지는 이유

이 사례는 가이드라인의 핵심 내용을 거의 그대로 보여준다.

### 8.1 AI는 블랙박스 도구가 아니다
비록 A 단계에서는 코드를 직접 거의 보지 않고 진행했지만,
그 과정에서 기술 이해와 판단 기준이 계속 쌓였다.

즉 결과만 받은 것이 아니라,
**내가 더 많이 이해하게 되었다**는 점이 중요하다.

### 8.2 A는 스펙 검증 개발이다
A 단계에서는 다음이 핵심이었다.

- 목표 스펙 정의
- 요구사항 수정
- 테스트 기준 정리
- 우선순위 조정
- 실제 동작 확인

즉 코드보다 스펙과 결과를 중심으로 전진했다.

### 8.3 B는 마이그레이션 개발이다
B 단계에서는 다음이 핵심이었다.

- 기존 프로젝트 분석
- 인터페이스와 컨벤션 확인
- 코드 리뷰 및 구조 파악
- 안전한 fallback 전략
- 실전 워크플로우에 맞춘 확장

즉 코드와 구조를 보며 편입을 진행했다.

### 8.4 B에서 AI가 항상 더 빠른 것은 아니다
이 사례는 AI가 매우 큰 도움을 줬지만,
B 단계의 본질은 결국 **프로젝트 이해와 판단**이라는 점도 보여준다.

즉 B에서는 AI를 맹목적으로 쓰는 것보다,
필요한 곳에 정확히 붙이는 것이 더 중요하다.

---

## 9. 이 사례에서 얻을 수 있는 실무 교훈

### 1) 외부 레포 재활용도 훌륭한 A 사례가 될 수 있다
처음부터 새로 만드는 것만 A가 아니다.
기존 레포를 가져와서 새로운 스펙으로 검증하는 것도 매우 좋은 A다.

### 2) 좋은 A는 단순 구현보다 좋은 질문에서 나온다
- 어떤 데이터셋으로 바꿀 것인가
- 어떤 출력 포맷으로 바꿀 것인가
- 어떤 품질을 목표로 할 것인가
- 어떤 고도화는 지금 미룰 것인가

이 질문들이 좋았기 때문에 A가 성공할 수 있었다.

### 3) 기술 이해가 쌓이면 AI 의존이 아니라 AI 활용이 된다
LPC, 포맷 변환, 학습 구조, 추론 비용 등 핵심 이해가 생기면서,
AI를 단순 대행자가 아니라 **기술 탐색 파트너**로 쓸 수 있게 되었다.

### 4) 좋은 B는 안전한 편입 전략을 가진다
새 기능을 넣을 때 기존 레거시 립싱크를 fallback으로 둔 전략은,
마이그레이션 개발에서 매우 좋은 판단이다.

### 5) 최종 가치는 “동작”이 아니라 “프로젝트 안에서 쓸 수 있는 상태”다
언리얼에서 돌아가는 데서 끝난 것이 아니라,
기존 플러그인 구조 안에 통합되고 에디터/시퀀서 워크플로우까지 확장되었다는 점에서
이 사례는 B까지 성공한 사례라고 볼 수 있다.

## 10. 한 문장 정리

이 사례는,
**AI 에이전트를 활용해 외부 기술을 빠르게 이해하고 스펙 검증 개발로 가능성을 확인한 뒤,
그 결과물을 기존 프로젝트에 맞게 마이그레이션 개발로 편입한 매우 좋은 실제 적용 사례**다.
