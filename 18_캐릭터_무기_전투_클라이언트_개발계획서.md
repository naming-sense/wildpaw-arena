## 4) 전투 로직 설계 (클라이언트 실행 규칙)

## 4-1. 입력 처리 파이프라인
1. 입력 샘플링 (`WASD`, `LMB`, `Q/E/R`)
2. `InputFrame` 생성 (`inputSeq`, `move`, `fire`, `skillFlags`)
3. 즉시 로컬 예측 적용
4. 서버 전송
5. authoritative snapshot 도착 시 리컨실리에이션

## 4-2. 무기 발사 로직

### 공통 단계
```text
canFire? (alive && !reloading && ammo>0 && fireIntervalReady)
 -> consumeAmmo
 -> local shot event (muzzle/vfx/sfx)
 -> recoil/spread apply
 -> send action command
```

### 거리 감쇠 공식(권장)
```text
if distance <= falloffStart: dmg = base
if distance >= falloffEnd:   dmg = base * minFalloff(예:0.62)
else 선형보간
```

### 재장전
- 재장전 시작 조건: `ammo == 0` 또는 수동 재장전 입력
- 재장전 중 발사 금지
- 서버 스냅샷 `reloadRemainingTicks`와 HUD 동기화

## 4-3. 스킬 로직 공통 템플릿

### Cast State Machine
```text
Idle -> CastWindup -> Resolve -> Recovery -> Idle
```

- `CastWindup`: 시전 시간(`castTimeMs`) 진행
- `Resolve`: 스킬 효과 적용(대시/존생성/버프)
- `Recovery`: 후딜(필요시)
- 중단 조건: 사망/스턴/루트(정의된 인터럽트 룰)

## 4-4. 스킬 아키타입별 구현 포인트

### A) Dash (예: 루미폭스 Q)
- 입력 방향 기반 거리 이동
- 대시 중 충돌 간소화(벽 관통 금지)
- 무적 시간은 로컬 연출만, 판정은 서버 확정

### B) Zone (예: 스팅크랫 Q/R)
- 존 중심 좌표 + 반경 + 지속시간
- 틱 단위 도트/감속 적용
- 존 중첩 룰(중복 감속 상한) 필요

### C) Shield/Heal (예: 밀키 E/R)
- 실드는 별도 값으로 관리 (`shieldHp`)
- 피해 우선순위: shield -> hp
- HoT는 tick 누적 오차 방지 위해 정수 tick 분배

### D) Channel Beam (예: 펄 Q/E)
- 채널 유지 조건: 입력 유지 + 라인 오브 사이트(서버 확정)
- 과열/리소스 소모와 연동

## 4-5. 상태이상 로직
- 효과 종류: `Slow`, `Root`, `Stun`, `DamageAmp`, `Shield`, `HealOverTime`
- 중첩 규칙:
  - 같은 그룹은 `max` 또는 `refresh` 정책 명시
  - 예: Slow는 최대 강도 1개만 적용, duration은 최대값 갱신

## 5) 히어로별 구현 체크리스트 (MVP 8)

## 5-1. 루미폭스/코랄 (Skirmisher)
- 이동기 + 재진입 버프 + 킬 관여 트리거
- 체크 포인트:
  - 처치 관여 판정 타이밍
  - 재진입 버프 중복 방지

## 5-2. 브루노/록혼 (Vanguard)
- 진입/넉백/넉업/방어 버프
- 체크 포인트:
  - 군중제어(CC) 저항/면역 상호작용
  - 전방 방어 방향 판정(도트곱 기반)

## 5-3. 스팅크랫 (Controller)
- 트랩 + 존 + 시야 방해
- 체크 포인트:
  - 트랩 설치/발동 중복 처리
  - 존 다중 생성 성능 관리

## 5-4. 밀키/펄 (Support)
- 힐/실드/구조형 궁극기

  - 저체력 보정 힐량 계산

## 5-5. 아이리스 (Striker)
- 표식 + 팀 피해 증폭 + 안정 화력
- 체크 포인트:
  - mark 소유권/지속시간
  - 팀 피해증폭 중복 상한

## 6) 네트워크 연동 상세 (서버 authoritative)

## 6-1. 프로토콜 매핑
- 입력: `InputPayload` / `ActionCommandPayload`
- 스냅샷: `SnapshotPayload.players[].PlayerState`
- 전투 이벤트: `CombatEventPayload`, `ProjectileEventPayload`

## 6-2. 예측/보정 알고리즘
```ts
// pseudo
onInput(frame):
  pending.push(frame)
  local = simulate(local, frame)
  send(frame)

onSnapshot(serverState):
  local = serverState.self
  replay = pending.filter(seq > serverState.lastProcessedInputSeq)
  for (f of replay) local = simulate(local, f)
  smoothIfNeeded(local, renderState)
```

## 6-3. 중복 이벤트/순서 역전 처리
- `serverTick` + `eventType` + `sourcePlayerId` + `targetPlayerId` 키로 디듀프
- 지연 도착 이벤트는 HUD만 보정, 위치는 최신 스냅샷 기준 유지

## 13) 최종 인수 기준 (클라이언트 관점)
- 8히어로 전투 루프가 30Hz에서 안정 동작
- 전투 UI와 authoritative snapshot 불일치 이슈 재현 불가
- 3v3 플레이테스트 20판 기준 치명/블로킹 버그 0
- 밸런스 데이터만 바꿔도(코드 수정 없이) 핵심 수치 조정 가능

> 즉, 이 계획서의 목표는 “컨셉을 코드로 옮길 때 생기는 모호함”을 제거하고,
> 클라이언트 팀이 **데이터-로직-테스트를 한 번에 굴릴 수 있는 개발 운영 체계**를 만드는 것이다.
