## 3-1. 현재 레포 구현 매핑 (실행 스캐폴드)

현재 `client/web`에는 아래 실구현 파일이 반영되어 있다.

- `src/netcode/*`
  - `RealtimeClient` (FlatBuffers C2S/S2C)
  - prediction/reconciliation/interpolation 유틸
- `src/gameplay/ecs/realtimeEcsRuntime.ts`
  - netcode + 입력 송신 + 보간/리컨실리에이션 결합 런타임
- `src/gameplay/three/threeCombatSceneAdapter.ts`
  - snapshot 동기화 + combat/projectile 이벤트 FX 반영
- `src/gameplay/three/threeRealtimeLoop.ts`
  - `requestAnimationFrame` 렌더 루프 + 입력 전송 루프 통합

즉, 문서 권장구조의 축약판이 아닌 **실행 가능한 연결 루프**가 이미 포함되어 있다.

## 4) 런타임 아키텍처

## 4-1. 루프 분리 원칙
- **Render Loop**: `requestAnimationFrame` (가변)
- **Simulation Loop**: 고정 스텝 `33.33ms(30Hz)`
- **Network Loop**: 수신 시점 기반 이벤트 처리

## 4-2. 업데이트 순서 (프레임 기준)
1. 입력 샘플링
2. 로컬 커맨드 생성 + 버퍼 저장
3. 로컬 예측 시뮬레이션
4. 서버 스냅샷 수신 처리(있을 때)
5. 리컨실리에이션
6. 원격 엔티티 보간
7. 렌더 갱신
8. HUD/디버그 오버레이 갱신

## 5) 입력 시스템 상세

## 5-1. 입력 매핑
- 이동: WASD
- 조준: 마우스 위치(월드 레이캐스트)
- 기본 사격: LMB
- 스킬: Q/E
- 궁극기: R
- 핑: MMB 또는 Alt+클릭

## 5-2. 커맨드 데이터 구조
```ts
export interface InputCommand {
  seq: number;           // 클라 증가 시퀀스
  clientTime: number;    // ms
  moveX: number;         // -1..1
  moveY: number;         // -1..1
  aimX: number;          // world space
  aimY: number;
  fire: boolean;
  skillQ: boolean;
  skillE: boolean;
  skillR: boolean;
}
```

## 5-3. 전송 정책
- 입력은 20~30Hz로 서버 전송
- 같은 틱에서 변화 없는 입력은 압축(전송 생략 또는 delta)
- seq 누락 대비 재전송 윈도우 유지(최근 N개)

## 6) 네트코드 구현 (핵심)

## 6-1. 상태 동기화 모델
- 서버 authoritative
- 클라 로컬 캐릭터는 prediction + reconciliation
- 타 플레이어는 interpolation buffer로 렌더

## 6-2. 로컬 예측
- 입력 즉시 로컬에 적용(이동/회전/애니메이션)
- 발사 연출(총구 화염, 사운드)은 즉시 재생 가능
- **최종 히트 판정은 서버 결과만 확정**

## 6-3. 리컨실리에이션 절차
1. 서버 스냅샷(ackSeq 포함) 수신
2. 로컬 상태를 ackSeq 기준 authoritative 상태로 교체
3. ackSeq 이후 커맨드를 순서대로 재적용(replay)
4. 오차가 임계치 초과 시 스냅 보정, 이하 시 smooth 보정

```ts
if (errorDistance > HARD_SNAP_THRESHOLD) {
  player.pos = serverPos;
} else {
  player.pos = lerp(player.pos, serverPos, 0.35);
}
```

## 6-4. 원격 보간
- 80~120ms 지연 버퍼 유지
- `t = renderTime - interpolationDelay`
- 스냅샷 A/B 사이 선형 또는 Hermite 보간

## 6-5. 패킷 손실 대응
- 스냅샷 누락 시 마지막 속도 기반 짧은 외삽(extrapolation, 최대 100ms)
- 장기 누락 시 순간이동 방지용 감쇠 스무딩 적용

## 6-6. 전투 HUD 동기화 필드 (현재 스키마)
- 플레이어 스냅샷에 아래 필드가 포함됨:
  - `ammo`, `maxAmmo`, `isReloading`, `reloadRemainingTicks`
  - `skillQCooldownTicks`, `skillECooldownTicks`, `skillRCooldownTicks`
  - `castingSkill`, `castRemainingTicks`
- 클라는 위 필드를 HUD/스킬 UI와 1:1로 연결하고,
  입력 즉시 예측 연출은 허용하되 최종 수치는 스냅샷 기준으로 확정한다.

## 8) 렌더링 파이프라인 (Three.js)

## 8-1. 카메라
- 탑뷰 고정 각도(35~45도), 부드러운 추적
- 화면 흔들림은 이벤트 기반 짧은 임펄스

## 8-2. 라이팅
- 주광 + 보조광 1~2개
- 그림자는 플레이어/주요 오브젝트만 활성
- 저사양 모드에서 shadow map 해상도 단계 축소

## 8-3. 머티리얼/이펙트
- PBR 간소화(roughness/metalness 규칙 고정)
- 투사체/스킬 이펙트는 GPU instancing 우선
- 투명 오브젝트 정렬 이슈 대비, FX 레이어 분리

## 8-4. LOD/컬링
- 캐릭터 LOD 2단계 + 거리 기반 애니메이션 업데이트 간격 조절
- 프러스텀 컬링 + 맵 섹터 단위 컬링

## 11) 성능 예산 (초기 기준)

- Draw Calls: 전투 시 평균 400 이하 (권장)
- CPU Main Thread: 프레임당 8ms 이하 목표
- GPU Frame Time: 10ms 이하 목표(60FPS 환경)
- 네트워크 처리: 프레임당 1ms 내외
- 메모리: 브라우저 탭 기준 700MB 이하 유지

### 최적화 우선순위
1. 이펙트 과다 렌더 절감
2. 불필요한 React 리렌더 제거
3. 오브젝트 풀링(투사체/파편/데미지 텍스트)
4. 문자열/JSON 파싱 최소화(바이너리 패킷)

## 16) 서버-클라 계약 체크리스트 (필수 동기화 항목)
- 엔티티 ID, 팀, 좌표/회전, 체력/실드, 상태이상
- 탄약, 재장전 상태, 스킬 쿨다운, 궁극기 충전량
- 목표물 점수/상태, 매치 타이머, 리스폰 타이머
- 판정 이벤트(피해/처치/어시스트/오브젝트 점수)

> 이 항목은 서버 변경 시 반드시 동시 검토해야 하며,
> 계약 불일치가 생기면 클라이언트 desync가 즉시 발생한다.
