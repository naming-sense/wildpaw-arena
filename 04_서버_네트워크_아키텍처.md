# 04. 서버 · 네트워크 아키텍처 (Three.js + C++ Asio)

## 1) 목표 품질 (체감 "무렉" 기준)
- **서버 틱**: 30Hz(기본), 경쟁 모드 60Hz 옵션
- **클라이언트 렌더**: 60FPS 목표
- **매치 내 지연 목표**
  - RTT p50 < 45ms
  - RTT p95 < 90ms
  - 지터 p95 < 15ms
- **서버 안정성**: 매치 서버 크래시율 < 0.1%/일

> 현실적으로 완전한 0ms는 불가능하므로, **예측/보간/복구**로 "렉 없는 체감"을 만드는 것이 핵심.

---

## 2) 전체 구성도

```text
[Web Client (Three.js)]
   ↕ WebSocket(WebTransport 가능시 우선)
[Edge Gateway]
   ├─ Auth Service
   ├─ Matchmaker
   ├─ Party/Presence
   └─ Realtime Room Allocator
              ↕
       [C++ Asio Game Room Servers]
              ↕
     [Redis (state cache) / Postgres (account, inventory)]
              ↕
     [Analytics Pipeline + Monitoring]
```

## 3) 클라이언트(Three.js) 구조

## 3-1. 주요 모듈
- `render-core`: Three.js scene/camera/light/postprocess
- `gameplay-ecs`: 엔티티/컴포넌트/시스템
- `netcode`: 패킷 encode/decode, 예측/보간/복구
- `input`: 키·마우스 샘플링, 입력 버퍼링
- `ui`: HUD/로비/결과

## 3-2. 프레임 루프 분리
- 렌더 루프: `requestAnimationFrame`(가변)
- 시뮬 루프: 고정 스텝(예: 33.3ms)
- 네트워크 루프: 스냅샷 수신 시점 독립 처리

## 3-3. 지연 보정
1. **Client-side Prediction**: 내 이동/사격 선반영
2. **Server Reconciliation**: 서버 authoritative 상태 수신 후 오차 재적용
3. **Interpolation Buffer**: 타 유저는 80~120ms 버퍼로 부드럽게 보간
4. **Lag Compensation**: 히트 판정 시 서버에서 과거 위치 재구성(리와인드)

## 4) C++ Asio 실시간 서버 설계

## 4-1. 기술 스택
- C++20
- standalone Asio (Boost 의존 최소화 가능)
- TLS/HTTP 업그레이드(초기 핸드셰이크)
- 직렬화: FlatBuffers 또는 커스텀 바이너리(VarInt + bitpack)

## 4-2. 프로세스 모델
- `gateway process`: 인증/라우팅
- `room worker process`: 실제 게임 시뮬레이션
- 룸 단위 격리(크래시 영향 최소화)

## 4-3. 스레드 모델(권장)
- NIC 수신/송신용 `io_context` 스레드풀 (코어 수 기준 N)
- 룸 시뮬레이션은 `strand` 또는 룸 전용 작업 큐로 직렬 보장
- DB/외부 I/O는 비동기 job queue 분리

## 4-4. 룸 시뮬레이션
- authoritative ECS
- 고정 틱 처리 순서:
  1) 입력 수집
  2) 이동/충돌
  3) 사격/투사체
  4) 스킬/버프
  5) 판정/점수
  6) 델타 스냅샷 생성
- 룸 최대 인원: 10명(5v5) 기준 설계

---

## 5) 네트워크 프로토콜

## 5-1. 채널 분리
- 신뢰 채널: 매치 시작/종료, 인벤토리, 확정 이벤트
- 비신뢰/저지연 채널: 위치, 에임, 투사체 상태

## 5-2. 패킷 최적화
- 델타 압축(변경 필드만 전송)
- 관심영역(Interest Management): 근접 엔티티 우선 전송
- 스냅샷 주기 조절(거리/가시성에 따라 10~30Hz)
- 압축 전송(zstd/lightweight)

## 5-3. 대역폭 목표(1인당)
- 수신: 10~25KB/s
- 송신: 8~20KB/s
- 10인 룸 기준 서버 egress 최적화 필수

---

## 6) 대규모 접속 대응 전략

## 6-1. 수평 확장
- 룸 서버는 **stateless에 가까운 세션 구조**로 수평 증설
- Matchmaker가 지역/핑/MMR 기반으로 룸 배치
- 오토스케일: 큐 길이, 활성 룸 수, CPU 사용률 기반

## 6-2. 샤딩/리전
- KR/JP/SG/US 등 지역 리전 분리
- 파티는 핑 상한(예: 120ms) 넘으면 경고/보정 룰 제공

## 6-3. 장애 격리
- 룸 프로세스 크래시 시 해당 매치만 종료/복구
- 게이트웨이 다중화 + 헬스체크 + 자동 재시작

---

## 7) 운영/관측(Observability)
- 메트릭: 틱 처리시간, RTT, 지터, 패킷 드랍률, 룸당 CPU/Mem
- 로그: 매치 이벤트 구조화(JSON)
- 트레이싱: 로그인→매칭→입장→경기 종료 전체 추적
- 알람: p95 RTT 급등, 특정 리전 드랍률 급증, 매치 생성 실패율

---

## 8) 보안/치트 대응
- 서버 authoritative 판정(클라 신뢰 금지)
- 입력 빈도/패턴 검증(매크로 탐지)
- 비정상 이동/발사 레이트 제한
- 리플레이 검증용 이벤트 로그(신고 대응)
- TLS + 토큰 기반 세션 검증

---

## 9) 구현 마일스톤(기술)
1. **M1 (4주)**: WebSocket 룸 입장 + 이동 동기화 + 기본 사격
2. **M2 (4주)**: 예측/복구/보간 + 히트 판정 안정화
3. **M3 (5주)**: 스킬/오브젝트/모드 룰 + 매치메이커
4. **M4 (4주)**: 확장 테스트(봇/부하) + 운영 지표 + 소프트런칭 준비

---

## 10) 부하 테스트 기준(출시 전)
- 동시 접속 CCU 5k, 10k, 30k 단계별 테스트
- 시나리오: 큐 폭주, 룸 서버 재시작, 리전 단절, 패킷 손실 5~10%
- 합격 기준:
  - 매치 시작 실패율 < 1%
  - 강제 종료율 < 0.5%
  - p95 서버 틱 지연 < 8ms(30Hz 기준)
