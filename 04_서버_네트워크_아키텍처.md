# 04. 서버 · 네트워크 아키텍처 (Three.js + C++ Asio)

> 목적: "무렉에 가까운 체감"을 만들기 위한 **실행 가능한 구현 명세**
> 
> 범위: 실시간 전투(3v3/5v5), 매칭/입장/경기종료 파이프라인, 운영 관측/복구

---

## 1) 목표 품질 (체감 "무렉" 기준)

- **서버 틱**: 30Hz(기본), 경쟁 모드 60Hz 옵션
- **클라이언트 렌더**: 60FPS 목표
- **매치 내 지연 목표**
  - RTT p50 < 45ms
  - RTT p95 < 90ms
  - 지터 p95 < 15ms
- **서버 안정성**: 매치 서버 크래시율 < 0.1%/일
- **입장 품질**
  - 매치 할당 → 룸 접속 완료 p95 < 3초
  - 매치 시작 실패율 < 1%

핵심 원칙:
- 서버 authoritative (판정은 서버가 최종)
- 클라이언트는 예측/보간으로 체감 품질 보정
- 룸 단위 격리로 장애 반경 최소화

---

## 2) 전체 구성도

```text
[Web Client (Three.js)]
   ↕ WebSocket (WebTransport 가능시 우선)
[Edge Gateway]
   ├─ Auth Service
   ├─ Matchmaker
   ├─ Party/Presence
   └─ Realtime Room Allocator
              ↕
      [C++ Asio Game Room Servers]
              ↕
    [Redis (state cache) / Postgres (account, inventory)]
              ↕
    [Analytics Pipeline + Monitoring]
```

구성 원칙:
- Edge는 세션/라우팅 중심, 룸 시뮬레이션은 별도 워커
- DB 직접 접근은 gameplay critical path에서 최소화
- 로그/분석은 비동기 파이프라인으로 분리

## 3) 클라이언트(Three.js) 구조

### 3-1. 주요 모듈
- `render-core`: scene/camera/light/postprocess
- `gameplay-ecs`: 엔티티/컴포넌트/시스템
- `netcode`: 패킷 encode/decode, 예측/보간/복구
- `input`: 키/마우스 샘플링, 입력 버퍼링
- `ui`: HUD/로비/결과

### 3-2. 루프 분리
- 렌더 루프: `requestAnimationFrame` (가변)
- 시뮬 루프: 고정 스텝(기본 33.3ms)
- 네트워크 루프: 수신 시점 독립 처리

### 3-3. 지연 보정 4종
1. **Client-side Prediction**: 내 이동/사격 선반영
2. **Server Reconciliation**: authoritative 상태로 오차 수정
3. **Interpolation Buffer**: 타 유저 80~120ms 버퍼 보간
4. **Lag Compensation**: 히트 판정 시 서버 리와인드

### 3-4. 필수 구현 디테일
- 입력 프레임에 `inputSeq`(증가값) 포함
- 클라는 최근 N프레임 입력(권장 120프레임) 링버퍼 저장
- authoritative snapshot의 `lastProcessedInputSeq` 기준으로 재적용
- 보간 버퍼는 네트워크 품질에 따라 동적(80~140ms) 조절

## 4) C++ Asio 실시간 서버 설계

### 4-1. 기술 스택
- C++20
- Boost.Asio + Boost.Beast (WebSocket)
- TLS/HTTP 업그레이드(초기 핸드셰이크)
- 직렬화: FlatBuffers (현재 구현), 필요 시 커스텀 바이너리 병행

### 4-2. 프로세스 모델
- `gateway process`: 인증/라우팅
- `room worker process`: 게임 시뮬레이션
- 룸 단위 격리 (한 룸 장애가 전체 장애로 전파되지 않도록)

### 4-3. 스레드 모델 (구현 반영)
- 네트워크 I/O: `io_context` 스레드풀 (`N`개, 실행 시 인자/코어 수 기반)
- `work_guard`: I/O 스레드가 조기 종료되지 않도록 유지
- 룸 시뮬: **전용 tick thread**(고정 주기 30Hz/옵션)
- 패킷 처리: 네트워크 스레드에서 디코드 후 **입력 워커 큐(`pendingInputs`)**에 적재, tick thread가 drain
- 외부 I/O(DB/로그): 추후 별도 비동기 큐 분리(현재 스캐폴드 단계)

### 4-4. 룸 틱 처리 순서 (authoritative)
1) 입력 큐 drain (`pendingInputs`)
2) 이동/충돌
3) 사격/투사체
4) 스킬/버프
5) 판정/점수
6) 델타 스냅샷 생성/브로드캐스트

- 룸 최대 인원: 10명(5v5)
- 목표 틱 budget(30Hz): **한 틱 33.3ms 중 시뮬 p95 < 8ms**

### 4-5. 세션 전송 순서 보장 (구현 반영)
- 세션은 `async_accept(make_strand(io_context))`로 생성
- 각 세션은 자체 `write_queue`를 유지
- `sendBinary()`는 세션 executor(strand)에 `post`
- 결과적으로 **세션별 송신 순서(in-order)** 보장 및 동시 write 충돌 방지

## 5) 네트워크 프로토콜

### 5-1. 채널 분리
- 신뢰 채널: 매치 시작/종료, 인벤토리, 확정 이벤트
- 비신뢰 저지연 채널: 위치, 에임, 투사체 상태

### 5-2. 패킷 최적화
- 델타 압축(변경 필드만)
- Interest Management(근접/가시 우선)
- 스냅샷 주기 가변(거리/가시성 기반 10~30Hz)
- 압축 전송(zstd/lightweight)

### 5-3. 대역폭 목표 (1인당)
- 수신: 10~25KB/s
- 송신: 8~20KB/s
- 10인 룸 기준 서버 egress 최적화 필수

### 5-4. 메시지 타입 (현재 구현)
- FlatBuffers `Envelope(seq, ack, ack_bits, payload union)`
- C2S payload
  - `HelloPayload`
  - `InputPayload`
  - `PingPayload`
- S2C payload
  - `WelcomePayload`
  - `SnapshotPayload(kind=Base|Delta)`
  - `EventPayload`

핵심 신뢰 메타:
- `seq(u32)`
- `ack(u32)`
- `ack_bits(u32)`

> 참고: 기존 `msgType/roomId/...` 커스텀 헤더 초안은 레거시로 유지하고, 실 전송은 FlatBuffers envelope 기반으로 통일.

## 11) 구현용 서비스 계약 (API/이벤트)

### 11-1. Gateway HTTP (초기)
- `POST /v1/auth/guest`
  - req: 디바이스/플랫폼 정보
  - res: access token, playerId
- `POST /v1/matchmaking/enqueue`
  - req: mode, partyId(optional), regionPreference
  - res: ticketId
- `POST /v1/matchmaking/dequeue`
  - req: ticketId

### 11-2. Gateway → Client 실시간 이벤트
- `match.found`: `matchId`, `roomEndpoint`, `roomToken`
- `match.cancelled`: reason
- `party.updated`: 파티원 상태 변경

### 11-3. Room 접속 핸드셰이크
1) Client `C2S_HELLO(roomToken, clientVersion)`
2) Server `S2C_WELCOME(playerSlot, serverTickRate, snapshotBase)`
3) Client부터 `C2S_INPUT` 스트림 시작

## 12) 룸 서버 내부 설계 (실행 관점)

### 12-1. 권장 디렉토리 구조 (실제 개발 레포 기준)
```text
server/
  gateway/
  matchmaker/
  room/
    src/
      net/
      sim/
      ecs/
      anti_cheat/
      snapshot/
    include/
shared/
  protocol/
  telemetry/
client/
  web/
    src/netcode/
```

### 12-2. 핵심 클래스
- `RoomSession`: 클라이언트 연결/전송/ack 상태
- `RoomSimulation`: authoritative world + tick 실행
- `InputBuffer`: 플레이어별 입력 프레임 큐
- `SnapshotBuilder`: baseline/delta 스냅샷 생성
- `InterestManager`: 엔티티 가시/거리 필터링

### 12-3. 룸 틱 의사코드
```cpp
void RoomSimulation::tick(uint32_t tickId) {
  collectInputs(tickId);               // timeout 입력은 마지막 입력 재사용
  applyMovement();                     // 물리/충돌
  processProjectiles();                // 발사체/충돌
  processSkillsAndBuffs();             // 스킬/상태이상
  resolveDamageAndScore();             // 체력/킬/점수
  auto events = drainAuthoritativeEvents();
  auto snapshots = snapshotBuilder.buildDelta(world, tickId);
  broadcaster.send(events, snapshots);
  antiCheat.scanFrame(world, tickId);
}
```

### 12-4. 입력 유실 처리
- `inputSeq` 누락 허용 범위: 최대 2~3프레임
- 초과 누락 시:
  - 마지막 입력 유지(짧게)
  - 이후 감쇠(가속/이동량 제한)
  - 장기 유실은 idle 처리

## 13) 클라이언트 Netcode 구현 지침

### 13-1. 로컬 플레이어
- 입력 즉시 로컬 시뮬 적용
- 전송 시 `inputSeq`와 `localTick` 포함
- 서버 snapshot 수신 시 `lastProcessedInputSeq`까지 롤백 후 재적용

### 13-2. 원격 플레이어
- snapshot 2개 이상 확보 시 보간
- 목표 보간 지연값: 100ms 기본
- 지터 증가 시 버퍼 자동 증가(최대 140ms)

### 13-3. 워프 방지
- 위치 오차가 임계값(예: 1.5m) 이하이면 스무딩
- 임계값 초과 시 단계적 보정(3~5프레임)
- 텔레포트 이벤트는 예외 처리(즉시 위치 반영)

## 18) 동접 확장 관점: 구현 점검 + 실측 요약

### 18-1. 점검 결과 (요청 항목)
- 멀티스레드 I/O: **적용됨** (`io_context` 스레드풀 + `work_guard`)
- 룸 틱 전용 스레드: **적용됨** (`tick thread` 고정 주기)
- 패킷 워커큐: **적용됨** (`pendingInputs` 큐 drain 후 틱 처리)
- 세션별 `strand + write_queue`: **적용됨** (송신 순서 보장)
- ack 기반 retransmit/유실 복구 큐: **적용됨** (`RoomSession` + reliable in-flight queue)
- Prometheus 지표 노출: **적용됨** (`metricsPort`, `/metrics`)
- 스페이셜 파티셔닝 기반 관심영역 필터: **적용됨** (grid rebuild + cell query)

### 18-2. 왜 동접 증가에 도움이 되는가
1. I/O 멀티스레드 → 소켓 read/write 병렬성 증가
2. 틱 전용 스레드 → 네트워크 burst가 시뮬 지연으로 전이되는 현상 완화
3. 입력 큐 → decode 경로와 시뮬 경로를 느슨하게 분리
4. 세션 strand/write_queue → 순서 꼬임/동시 write 충돌 방지
5. ack 기반 retransmit → 패킷 유실 시 중요 메시지 복구 가능
6. 스페이셜 파티셔닝 → 관심영역 계산 비용 감소, 팬아웃 효율 개선
7. Prometheus → 병목(큐 깊이, tick overrun, retransmit)을 즉시 가시화

### 18-3. 로컬 실측(참고)
- 동일 머신/로컬호스트 벤치(입력 20Hz/클라당, ack 추적 포함) 관측값:
  - 150 clients: `snapshotPerSec ≈ 4064`
  - 200 clients: `snapshotPerSec ≈ 3136`
- 단일 루프 초안 대비 피크 처리량 상승
  - 대략 `~2.3k/s` → `~4.0k/s` 수준

> 주의: 위 수치는 로컬 synthetic 벤치이며, 실제 운영 수용량은 네트워크 조건/리전 거리/CPU 타입에 따라 달라진다.
